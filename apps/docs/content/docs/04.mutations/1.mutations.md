Call a Convex mutation from a component and get reactive state for free — pending, error, last result — without wiring separate refs or try/catch blocks.

```vue [app/pages/todos.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const title = ref('')
const createTodo = useConvexMutation(api.todos.create)

async function handleSubmit() {
  await createTodo({ title: title.value })
  title.value = ''
}
</script>

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="title" required />
    <UButton type="submit" :loading="createTodo.pending.value">Add todo</UButton>
  </form>
  <p v-if="createTodo.error.value">{{ createTodo.error.value.message }}</p>
</template>
```

Create the composable once near the top of `<script setup>`, then call it from the submit handler. The returned callable carries the mutation state, which is why you keep it around instead of inlining the call.

## What you get back

The returned function carries reactive state. Read it directly in templates (`pending.value` / `error.value`):

| Key       | What it holds                                                 |
| --------- | ------------------------------------------------------------- |
| `status`  | `idle`, `pending`, `success`, `error`                         |
| `pending` | Shorthand for `status === 'pending'`                          |
| `data`    | The last successful result                                    |
| `error`   | The last error (Trellis call error, with code when available) |
| `reset()` | Clear the tracked state                                       |

No need for a separate `isLoading` ref.

## Side effects: `onSuccess` / `onError`

For toasts, redirects, or navigation after a write, use the callbacks instead of wrapping the call in try/catch:

```ts [app/pages/projects.vue]
const createProject = useConvexMutation(api.projects.create, {
  onSuccess: (project) => {
    toast.add({ title: 'Project created', color: 'success' })
    navigateTo(`/projects/${project._id}`)
  },
  onError: (error) => {
    toast.add({ title: 'Could not create', description: error.message, color: 'error' })
  },
})
```

Both callbacks receive the args passed to the mutation, so you can include them in the toast or log.

## Error handling

Failures come back as Trellis call errors. When the backend attaches a structured code, you can branch on it:

```ts [composables/useCreateProject.ts]
const createProject = useConvexMutation(api.projects.create, {
  onError: (error) => {
    if (error.code === 'QUOTA_EXCEEDED') {
      navigateTo('/billing/upgrade')
      return
    }
    toast.add({ title: error.message, color: 'error' })
  },
})
```

::warning
In development, Trellis warns if a mutation errors and nobody reads `.error.value` — silent mutation failures are the easiest way to ship a broken form that looks like it worked. Don't suppress the warning; handle the error.
::

## Latest-call-wins state

If the user fires the same mutation twice quickly, the tracked state follows the **latest** call — not whichever request settles last. The earlier request might still resolve in the background, but your UI won't flicker back to its old state.

That's what you want for rapid-click scenarios (retry, spam-click, double-submit after a slow network hiccup).

## When to call `reset()`

Clear the state when the UI session ends, so stale errors or success banners don't bleed into the next flow:

```ts [app/components/CreateTodoDialog.vue]
function handleClose() {
  createTodo.reset()
  isOpen.value = false
}
```

Common triggers: closing a dialog, retrying after an error, dismissing a success banner before the next action.

## Common pitfalls

- Using an [action](/docs/mutations/actions) when the work is really a normal state-changing mutation.
- Building separate loading refs when `pending.value` already covers it.
- Ignoring the dev-mode warning about unhandled errors.

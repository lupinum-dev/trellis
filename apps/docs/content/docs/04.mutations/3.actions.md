Call a Convex action from a component when the work involves an external service, a longer-running job, or side effects that don't fit a normal mutation. The API shape mirrors [`useConvexMutation`](/docs/mutations/mutations) — same reactive state, same callbacks — so your UI code doesn't branch.

```ts [app/components/InviteButton.vue]
const sendInvite = useConvexAction(api.members.sendInvite, {
  onSuccess: () => toast.add({ title: 'Invite sent', color: 'success' }),
  onError: (err) => toast.add({ title: 'Invite failed', description: err.message, color: 'error' }),
})

async function handleInvite() {
  await sendInvite({ email: 'someone@example.com' })
}
```

Like mutations: the composable is synchronous. Await the returned call, not the composable itself.

## What stays the same

Everything about the call surface:

| Key                                                 | Same as mutations |
| --------------------------------------------------- | ----------------- |
| `status` / `pending` / `data` / `error` / `reset()` | Yes               |
| `onSuccess` / `onError` callbacks                   | Yes               |
| Latest-call-wins state                              | Yes               |
| Dev warning on unhandled errors                     | Yes               |

If you know the mutation shape, you already know the action shape.

## What's different

Not the client API — the **backend function type**. Actions run in a different Convex execution environment from queries and mutations: they can call external APIs, they don't run in a transaction, and they're the right home for side-effecting work.

Reach for `useConvexAction` when:

- calling an external service (email, Stripe, another HTTP API);
- running a longer job that shouldn't block a mutation transaction;
- producing side effects that shouldn't roll back with query-side state.

::warning
Don't use an action just because the flow feels "important" or the button says "Send". If the work is a normal state change inside your data model, it's a mutation. Overusing actions weakens the guarantee that mutations are transactional.
::

## Common pitfalls

- `await useConvexAction(...)` — sync composable, async call.
- Picking an action because it sounds heavier, when the work is really a mutation.
- Treating actions as the default write path. They aren't.

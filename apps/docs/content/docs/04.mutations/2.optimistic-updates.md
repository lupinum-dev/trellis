Change the visible UI the moment the user clicks, before the server round-trip resolves. Drag a card to a new column and watch it stay there — no flicker back while the mutation is pending.

```ts [app/components/TaskCard.vue]
const moveTask = useConvexMutation(api.tasks.moveToColumn, {
  optimisticUpdate: (ctx, args) => {
    ctx
      .query(api.tasks.listByProject, { projectId: props.projectId })
      .update(
        (tasks) => tasks?.map((t) => (t._id === args.id ? { ...t, status: args.status } : t)) ?? [],
      )
  },
})
```

The live query is still the source of truth. If the server rejects the move, the optimistic change rolls back. If the server accepts it, the live update overwrites your guess with real data.

## The mental model

An optimistic update is not "pretend it worked forever." It's four steps, and you only own the first one:

1. You update the local query state immediately.
2. The mutation runs on the server.
3. The live subscription returns fresh truth.
4. The UI corrects itself if your guess was wrong.

Keep the optimistic patch small, and keep it tied to what the mutation is actually going to change. If the real server result involves computed fields or sibling tables the UI also reads, you're asking the optimistic update to lie about things it can't predict — which reads worse than a brief spinner would.

## What `ctx` gives you

| Builder                        | Use when                                                    |
| ------------------------------ | ----------------------------------------------------------- |
| `ctx.query(fn, args)`          | Updating a single regular `useConvexQuery`                  |
| `ctx.paginatedQuery(fn, args)` | Updating a `useConvexPaginatedQuery` list                   |
| `ctx.matchQuery(fn)`           | Every active arg combination of that query needs the update |
| `ctx.matchPaginatedQuery(fn)`  | Paginated equivalent of `matchQuery`                        |

Pick the narrowest one. "Every active query" looks tempting but is rarely what the UI actually needs, and it's where optimistic updates go from helpful to confusing.

## Regular query — insert at top

```ts [app/components/AddNote.vue]
const addNote = useConvexMutation(api.notes.add, {
  optimisticUpdate: (ctx, args) => {
    ctx
      .query(api.notes.list, {})
      .update((notes) =>
        notes ? [{ _id: 'temp', ...args }, ...notes] : [{ _id: 'temp', ...args }],
      )
  },
})
```

## Paginated query — insert without touching cursors

```ts [app/components/CreateTask.vue]
const createTask = useConvexMutation(api.tasks.create, {
  optimisticUpdate: (ctx, args) => {
    ctx.paginatedQuery(api.tasks.listByProject, { projectId: args.projectId }).insertAtTop({
      _id: 'temp',
      title: args.title,
      status: 'backlog',
    } as never)
  },
})
```

The builder handles cursor math. You describe the visible change, not the pagination state.

## Always keep error handling

Optimistic UI without error handling is worse than slow UI — the user sees their change, thinks it worked, and only finds out it didn't when something _else_ breaks. Always pair it with an `onError`:

```ts [app/components/TaskCard.vue]
const moveTask = useConvexMutation(api.tasks.moveToColumn, {
  optimisticUpdate: (ctx, args) => {
    /* ... */
  },
  onError: (error) =>
    toast.add({ title: 'Could not move task', description: error.message, color: 'error' }),
})
```

::caution
The optimistic patch is UX only. It does nothing on the server — the actual mutation still runs through [guards and authorization](/docs/concepts/glossary#authorize). Don't think of optimistic updates as a security or validation shortcut; they're purely visual.
::

## Good fit, bad fit

| Good                              | Bad                                          |
| --------------------------------- | -------------------------------------------- |
| Toggle completion                 | Backend rejects this mutation often          |
| Move card between columns         | Mutation changes many distant UI surfaces    |
| Insert new item into visible list | Final result is heavily computed server-side |
| Remove item from current page     | Server rewrites fields you can't predict     |

When in doubt, skip the optimistic update. A short spinner beats a lie.

## Common pitfalls

- Guessing at computed fields the server actually sets (timestamps, totals, derived status). Your guess will disagree with reality for a frame and look buggy.
- Using `matchQuery` when one `query` update would do.
- Treating optimistic UI as a substitute for real error handling.
- Not cleaning up the temp `_id` — if your list key is `_id`, a real ID arriving will remount the row. Use a stable key where it matters.

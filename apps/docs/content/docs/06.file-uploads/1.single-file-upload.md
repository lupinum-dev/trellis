Upload one file from the browser with progress, type checks, and size limits. You write the form; Trellis handles the mutation call, the XHR, the progress events, and the resulting `storageId`.

```vue [app/components/AvatarUpload.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const {
  upload,
  pending,
  progress,
  data: storageId,
  error,
} = useConvexUpload(api.files.generateUploadUrl, {
  allowedTypes: ['image/*'],
  maxSizeBytes: 5_000_000,
})

async function handleFile(event: Event) {
  const file = (event.target as HTMLInputElement).files?.[0]
  if (file) await upload(file)
}
</script>

<template>
  <input type="file" accept="image/*" @change="handleFile" />
  <progress v-if="pending" :value="progress" max="100" />
  <p v-if="error">{{ error.message }}</p>
</template>
```

## What's happening under the hood

The composable runs a four-step flow; you only write step one (the Convex mutation that returns an upload URL):

1. Calls your Convex mutation to get a signed upload URL.
2. POSTs the file via XHR, emitting progress events.
3. Exposes reactive state: `pending`, `progress`, `data`, `error`.
4. Returns the resulting `storageId` — the value you store on your document.

::warning
Your Convex mutation **must return a string** (the upload URL). If it returns something else, the upload fails immediately with a confusing error. Return the URL directly, not wrapped in an object.
::

## Client-side validation

`allowedTypes` and `maxSizeBytes` reject files before the upload starts — useful UX, not a security boundary:

```ts [app/components/AvatarUpload.vue]
useConvexUpload(api.files.generateUploadUrl, {
  allowedTypes: ['image/*', 'application/pdf'],
  maxSizeBytes: 5_000_000,
})
```

::caution
Client-side checks are advisory. A hostile client will strip them. Validate file type and size on the server side too — these UI limits exist so well-behaved users get fast feedback, not so unsafe files can't reach your storage.
::

## Cancel and reset

Call `reset()` to abort the in-flight upload and clear tracked state (status, error, progress, `storageId`):

```ts [app/components/AvatarUpload.vue]
const { upload, reset } = useConvexUpload(/* ... */)

function handleCancel() {
  reset()
}
```

When the component unmounts mid-upload, Trellis aborts automatically. You don't have to wire lifecycle cleanup.

## Persist the storage id

The `storageId` from the upload is just an opaque string. Your app has to store it — usually by calling a second mutation that attaches it to the document:

```ts [app/components/AvatarUpload.vue]
const setAvatar = useConvexMutation(api.users.setAvatar)

async function handleFile(event: Event) {
  const file = (event.target as HTMLInputElement).files?.[0]
  if (!file) return

  const id = await upload(file)
  await setAvatar({ storageId: id })
}
```

Forgetting this step is the most common upload bug — the file lands in storage, nothing references it, and it eventually gets cleaned up.

## Common pitfalls

- Returning something other than a raw string from the upload-URL mutation.
- Treating `allowedTypes` as the entire security model.
- Uploading successfully and never attaching the `storageId` to a document.
- Starting a new single-file upload while one is still pending — the call rejects. Wait for the current upload to finish, or call `reset()` first.

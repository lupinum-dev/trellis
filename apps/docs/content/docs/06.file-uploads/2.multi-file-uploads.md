Queue multiple files with bounded concurrency, per-item progress, cancel-per-item, and an aggregate status you can show in one banner. Same `useConvexUpload` composable — pass multiple files and it switches to queue mode.

```vue [app/components/BulkUploader.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const { upload, items, cancelItem, cancelAll, clearFinished, progress } = useConvexUpload(
  api.files.generateUploadUrl,
  {
    maxConcurrent: 3,
    continueOnError: true,
  },
)

async function handleFiles(event: Event) {
  const files = (event.target as HTMLInputElement).files
  if (!files?.length) return

  try {
    await upload(Array.from(files))
  } catch {
    // Per-item errors are visible in `items`; decide here whether to show a banner.
  }
}
</script>

<template>
  <input type="file" multiple @change="handleFiles" />
  <progress :value="progress" max="100" />
  <ul>
    <li v-for="item in items" :key="item.id">
      {{ item.file.name }} — {{ item.status }}
      <button v-if="item.status === 'pending'" @click="cancelItem(item.id)">Cancel</button>
    </li>
  </ul>
</template>
```

## What you get back

| Key                          | What it is                                                   |
| ---------------------------- | ------------------------------------------------------------ |
| `items`                      | Array of `{ id, file, status, progress, error, storageId }`  |
| `upload(fileOrFiles, args?)` | Upload one `File`, or enqueue a `File[]` / `FileList` batch  |
| `cancelItem(id)`             | Cancel one queued or in-flight upload                        |
| `cancelAll()`                | Cancel everything pending                                    |
| `clearFinished()`            | Drop completed/errored/cancelled rows from the visible queue |
| `progress`                   | Aggregate 0-100 across the whole queue                       |

Per-item `status` is one of: `queued`, `pending`, `success`, `error`, `cancelled`.

A single `File` uses single-upload mode and resolves with one `storageId`. A
`File[]` or `FileList` uses queue mode and resolves with all successful
`storageId`s after the batch settles.

## Concurrency is explicit

`maxConcurrent` caps how many uploads run in parallel. Resolution order:

1. The value you pass to `useConvexUpload(...)` wins.
2. Otherwise, module config: `trellis.upload.maxConcurrent`.
3. Otherwise, Trellis's default.

Don't leave concurrency implicit on a page where users might select hundreds of files — that's how browsers pin a CPU core on the XHR thread.

## Per-file mutation args

`useConvexUpload` passes the same `args` value to each file in a batch:

```ts [app/components/BulkUploader.vue]
const { upload } = useConvexUpload(api.files.generateUploadUrl)

await upload(files, { folder: 'invoices' })
```

That means a selected batch should represent one upload intent: one album, folder, size class, or other shared metadata shape. If each file needs distinct mutation args, split the UI into separate batches or build an app-specific queue around your upload-url mutation. The public `useConvexUpload` API intentionally keeps per-item metadata out of the high-level batch path.

## Continue or halt on error

`continueOnError` defaults to `true`, but it is still a product decision — be deliberate:

- `continueOnError: true` — the other uploads keep going when one fails. Right for bulk imports where partial success is useful.
- `continueOnError: false` — one failure stops the rest. Right for "all files are part of one transaction" UX (attaching a receipt bundle to an expense, say).

::tip
Show the user which mode they're in. If files silently stop uploading after one failure, they assume everything succeeded until they look.
::

Even with `continueOnError: true`, the batch promise rejects with an
`AggregateError` when one or more files fail. Await it so the component can show
a batch-level error while the per-file rows remain visible in `items`.

## Cancel vs. clear

Two different operations:

- **`cancelItem(id)` / `cancelAll()`** — abort in-flight or queued uploads. Items move to `cancelled` status and stay visible.
- **`clearFinished()`** — remove completed, errored, and cancelled items from the UI. The items don't "uncancel" — they just stop appearing.

Don't auto-clear failed items. Users often want to inspect the errors before dismissing them.

## Common pitfalls

- Running a `for` loop of single-file `upload()` calls instead of `upload(files)` — loses concurrency control and aggregate progress.
- Calling `upload(Array.from(files))` without `await` / `catch` — failed batches then become unhandled promise rejections.
- Forgetting `maxConcurrent` can come from module config, so you "set it once" but the page uses a stale default.
- `clearFinished()` too eagerly — users lose the error list they were about to read.
- Passing one mixed batch when each file needs distinct upload-url args. Split the files by shared metadata instead.

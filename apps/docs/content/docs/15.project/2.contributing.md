Get Trellis building locally and know where each part of the surface lives. The repo-root [CONTRIBUTING.md](https://github.com/lupinum-dev/trellis/blob/main/CONTRIBUTING.md) is the canonical guide — this page is the short orientation.

## Local start

```bash [terminal]
pnpm install
pnpm run dev:prepare
pnpm run dev
```

That gives you the module, the example apps, and the maintainer apps in one workspace.

## Where the code lives

| Path        | What's in it                                                        |
| ----------- | ------------------------------------------------------------------- |
| `src/`      | Module, runtime, MCP, server, and testing surfaces                  |
| `examples/` | Runnable example apps                                               |
| `tests/`    | Root unit, Nuxt, server, browser, e2e, fixtures, and support suites |
| `apps/`     | Runnable maintainer apps, including the docs site and harness       |
| `labs/`     | Experimental or archived example material                           |

## When docs should change in the same PR

Docs travel with the code whenever the change touches anything a reader relies on:

- Public runtime behavior.
- Config shape.
- Setup or wiring flow.
- Permissions or trust-boundary semantics.
- API surface or examples that docs reference.

::tip
If a PR changes a public signature but leaves docs stale, a follower will copy the old shape and file a bug. Update the page in the same PR — even a two-line change counts.
::

## Preview and validation

| Command                          | What it does                                          |
| -------------------------------- | ----------------------------------------------------- |
| `pnpm --dir apps/docs dev`       | Preview docs locally                                  |
| `pnpm check:docs:links`          | Validate internal doc links                           |
| `pnpm check:docs:api-surface`    | Catch drift between docs and the public API           |
| `pnpm run check:publish-surface` | Validate published exports and declaration specifiers |
| `pnpm run release:verify`        | Full release-style verification gate                  |

Run these before opening the PR — CI runs them too, but catching drift locally is faster.

## Public surface checklist

When a change touches package exports, generated aliases, auto-imports, starter
contracts, auth/MCP/server helper APIs, bridge package APIs, or release scripts:

1. Update the code and the generated/public docs in the same PR.
2. Check `package.json` exports, [API surface](/docs/api-reference/api-surface),
   and `tests/unit/package-subpath-exports.test.ts` together.
3. Run `pnpm run check:publish-surface` plus the relevant focused tests.
4. Run `pnpm run release:verify` before release handoff.

Do not maintain a second handwritten export inventory. The generated API surface
page is the docs source readers should trust.

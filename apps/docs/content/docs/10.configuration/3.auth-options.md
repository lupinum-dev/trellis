Reference for `trellis.auth`. Two common shapes: a boolean for enable/disable, or an object when you need to configure specific behavior.

```ts [nuxt.config.ts]
trellis: {
  auth: {
    enabled: true,
    route: '/api/auth',
    routeProtection: {
      redirectTo: '/auth/signin',
      preserveReturnTo: true,
    },
    unauthorized: {
      enabled: false,
      redirectTo: '/auth/signin',
      includeQueries: false,
    },
    trustedOrigins: [],
    skipAuthTokenFetchRoutes: [],
    cache: { enabled: false, ttl: 60 },
    proxy: {
      maxRequestBodyBytes: 1_048_576,
      maxResponseBodyBytes: 1_048_576,
    },
  },
}
```

## Shorthand

| Value         | Effect                |
| ------------- | --------------------- |
| `auth: true`  | Enable with defaults  |
| `auth: false` | Disable auth entirely |

## Option reference

| Option                             | Default                                                   | Purpose                                             |
| ---------------------------------- | --------------------------------------------------------- | --------------------------------------------------- |
| `enabled`                          | `false` when `auth` is omitted; object defaults to `true` | Master switch                                       |
| `route`                            | `/api/auth`                                               | Better Auth proxy route                             |
| `routeProtection.redirectTo`       | `/auth/signin`                                            | Default route-middleware sign-in target             |
| `routeProtection.preserveReturnTo` | `true`                                                    | Append return path when middleware redirects        |
| `unauthorized.enabled`             | `false`                                                   | Redirect on structured unauthorized failures        |
| `unauthorized.redirectTo`          | `/auth/signin`                                            | Unauthorized redirect target                        |
| `unauthorized.includeQueries`      | `false`                                                   | Include query failures in unauthorized redirects    |
| `trustedOrigins`                   | `[]`                                                      | Extra origins allowed through proxy CORS validation |
| `skipAuthTokenFetchRoutes`         | `[]`                                                      | Routes that skip auth token fetches                 |
| `cache.enabled`                    | `false`                                                   | SSR auth token cache on/off                         |
| `cache.ttl`                        | `60`                                                      | Cache TTL (seconds, clamped 1–60)                   |
| `proxy.maxRequestBodyBytes`        | `1_048_576`                                               | Request body size limit                             |
| `proxy.maxResponseBodyBytes`       | `1_048_576`                                               | Response body size limit                            |

## Important distinctions

::caution
`skipAuthTokenFetchRoutes` skips the _auth token fetch_ on specific routes. It isn't a replacement for `definePageMeta({ skipAuthTokenFetch: true })`, which skips token fetches for one page at a time. They solve the same concern at different scopes.
::

::note
At the module level, Trellis defaults to no auth. Use `auth: true` or `auth: { enabled: true }` when you want the Better Auth path; the `personal`, `workspace`, and `workspace-mcp` starters configure this explicitly.
::

::note
`cache` is opt-in. It trades a small correctness window (up to `ttl` seconds of
stale auth/session/JWT state) for fewer token-exchange round-trips per SSR
request. It does not cache the access-context projection itself. Right for
traffic-heavy deployments, wrong for local dev where you want to see auth
changes immediately.
::

`cache.ttl` is normalized into the safe 1–60 second range — passing `3600` clamps to `60`.

The cache uses Nitro storage namespace `cache:convex:auth`. On a multi-instance
deployment, configure that namespace to a shared backend such as Redis if you
expect cache hits across instances. The memory driver is per process.

## Rate-limit boundary

Better Auth handles **auth endpoint** rate limiting. That protects flows like sign-in and password reset. It is not a general limiter for every Convex function in your app.

If you need:

- MCP ingress throttling, use Trellis MCP tool `rateLimit`
- business quotas inside Convex, keep that policy in the consumer app

See [Rate-limiting layers](/docs/concepts/rate-limiting-layers) for the full model.

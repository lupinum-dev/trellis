Reference for `trellis.mcp`. These are module-level flags only. Tool definitions, runtime auth, actingFor, and rate-limit stores live in the MCP runtime files.

```ts [nuxt.config.ts]
trellis: {
  mcp: {
    name: 'My App',
    sessions: false,
  },
}
```

## Option reference

| Option     | Default | Purpose                   |
| ---------- | ------- | ------------------------- |
| `name`     | `''`    | Name shown to MCP clients |
| `sessions` | `false` | Enable MCP session state  |

## What is not configured here

MCP rate limiting is no longer configured through `trellis.mcp`. If a tool uses `rateLimit`, the runtime decides how that tool is enforced.

Use the MCP runtime for that:

```ts [server/mcp/runtime.ts]
import Redis from 'ioredis'
import { createRedisMcpRateLimitStore, defineMcpApp } from '@lupinum/trellis/mcp'

const redis = new Redis(process.env.REDIS_URL!)

export const mcpRuntime = defineMcpApp({
  rateLimitStore: createRedisMcpRateLimitStore({
    client: redis,
  }),
  // ...
})
```

Without `rateLimitStore`, Trellis falls back to process-local memory. That is fine for local work, tests, and single-process throwaway environments. It is not production-grade distributed enforcement.

For distributed MCP enforcement, the documented first-party atomic path is `createRedisMcpRateLimitStore(...)`.

::note
This limiter protects the **MCP tool ingress**. It does not rate-limit every Convex endpoint in your app. For business quotas inside Convex, keep the policy in the consumer app. See [Rate-limiting layers](/docs/concepts/rate-limiting-layers).
::

::note
`trellis doctor` can statically recognize the first-party Redis helper. If you wire a custom store, doctor treats it as unverified and you own the atomicity review.
::

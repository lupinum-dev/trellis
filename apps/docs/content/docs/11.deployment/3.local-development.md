Two common local paths: running an example app, or developing the module itself through the internal harness. They share most of the env layout but have different launchers.

| Mode              | Use it for                                              | Entry point                      |
| ----------------- | ------------------------------------------------------- | -------------------------------- |
| Example app       | Understanding or demoing a consumer pattern             | `cd examples/<name> && pnpm dev` |
| Generated starter | Starting a new app against local Convex                 | `pnpm dev:local`                 |
| Internal harness  | Feature development, auth/MCP verification, regressions | `pnpm dev` from repo root        |

## Example app flow

```bash [terminal]
cd examples/01-public-todo
cp .env.example .env.local  # if the example needs app-owned env
pnpm install
pnpm dev
```

In examples, `pnpm dev` starts a local Convex deployment and injects local Convex URLs for Nuxt. That means:

- `CONVEX_URL` and `CONVEX_SITE_URL` are _launcher-owned_.
- Auth secrets and provider credentials are still app-owned when the example needs them.

## Generated starter flow

```bash [terminal]
pnpm install
pnpm dev:local
```

Generated starters use the same local Convex Vite plugin pattern as the harness. `pnpm dev:local` points Trellis at `http://127.0.0.1:3210` and starts the local Convex backend. Use `pnpm dev` with `pnpm convex:dev` when you want a hosted Convex dev deployment instead.

## Internal harness flow

The harness lives in `apps/harness/`. Root-level commands:

| Command                | Does                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| `pnpm dev:prepare`     | Prepare module types and the harness workspace without starting Nuxt |
| `pnpm dev`             | Harness against default env                                          |
| `pnpm dev:local`       | Force harness onto local Convex ports                                |
| `pnpm dev:local:reset` | Same, but reset backend state first                                  |
| `pnpm dev:build`       | Build the harness without starting it                                |

For the internal harness, `USE_LOCAL_CONVEX=true` switches Nuxt config onto local Convex ports.

## Local env layout

| File           | Purpose                                           |
| -------------- | ------------------------------------------------- |
| `.env.example` | Checked-in example defaults                       |
| `.env.local`   | Shared local runtime file used by Convex and Nuxt |

The launcher rewrites launcher-owned keys in `.env.local` when it starts local services; it preserves app-owned values already there.

## Typical local defaults

```bash [.env.local]
CONVEX_URL=http://127.0.0.1:3210
CONVEX_SITE_URL=http://127.0.0.1:3211
SITE_URL=http://localhost:3000
```

## Running a dedicated local Convex backend

When you need a dedicated backend for auth, MCP, or E2E:

```bash [terminal]
cd apps/harness
pnpm exec convex dev --local
```

If auth setup is missing locally:

```bash [terminal]
pnpm exec convex env set SITE_URL http://localhost:3000 --env-file .env.local
pnpm exec convex env set BETTER_AUTH_SECRET <strong-random-secret> --env-file .env.local
```

## Useful root commands during local work

| Command                      | When                                           |
| ---------------------------- | ---------------------------------------------- |
| `pnpm check:cli`             | Touching CLI behavior                          |
| `pnpm check:publish-surface` | Touching package exports or declaration output |
| `pnpm test:types`            | Type-level regressions                         |
| `pnpm test:contracts`        | Public API surfaces                            |
| `pnpm test:internals`        | Internal harness logic                         |
| `pnpm test:nuxt`             | Focused Nuxt runtime debugging                 |
| `pnpm test:browser`          | Focused browser component debugging            |
| `pnpm test`                  | Full test suite                                |
| `pnpm lint`                  | Linting                                        |
| `pnpm docs:api-surface`      | Public API surfaces changed                    |

::note
`pnpm dev:prepare` is the only supported root-level prepare command. It prepares types and the harness workspace only. It does not rebuild the published package or CLI. If you need fresh package artifacts, run `pnpm build`.
::

## Common pitfalls

- Editing launcher-owned local Convex URLs by hand in a repo example. Let the launcher own them.
- Using the internal harness when an example app would be a clearer reproduction.
- Treating `pnpm dev:prepare` as a full rebuild.
- Treating local defaults as the recommended production deployment shape.

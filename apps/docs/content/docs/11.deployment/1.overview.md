A deployed Trellis app has three fixed parts and a couple of optional ones. This page sketches the shape; [Production checklist](/docs/deployment/production-checklist) is the actual pre-launch gate.

## Core production shape

| Piece                 | Always             | Why                                  |
| --------------------- | ------------------ | ------------------------------------ |
| One Nuxt deployment   | Yes                | Serves SSR, client, and Nitro routes |
| One Convex deployment | Yes                | Business layer, data, handlers       |
| Better Auth env       | When auth is on    | Token exchange depends on it         |
| Trusted-caller secret | For webhooks / MCP | Server-to-server identity            |

## Minimum env concerns

- `CONVEX_URL` points at the correct Convex deployment.
- `SITE_URL` matches the public app origin.
- `BETTER_AUTH_SECRET` is set when auth is enabled.
- `CONVEX_SITE_URL` is set when auth token exchange can't be derived (custom HTTP Actions domain).

See [Environment variables](/docs/configuration/environment-variables) for the detail.

## Advanced production concerns

Only some apps need these:

- `CONVEX_IDENTITY_FORWARDING_KEY` — server-owned signing secret for signed
  `_trellisForwarding` envelopes on webhook, MCP, server, or bridge traffic.
- MCP setup — enable the surface with `trellis.mcp`, then wire auth, tools, rate-limit storage, and a distributed confirmation store for destructive `confirmationMode: 'transport'` tools in `server/mcp/runtime.ts`.
- SSR auth cache backed by real storage — in-memory Nitro storage won't scale past one node.

## Things to keep straight

::warning
Browser auth and identity-forwarding auth are different trust paths. Shipping a webhook handler with `auth: 'none'` because "it's a webhook and it worked locally" is how integration handlers end up doing more than any human user can. See [Webhooks and identity forwarding](/docs/server-side/webhooks-and-identity-forwarding).
::

- Local-development shortcuts don't carry to production. If auth worked locally through an injected `CONVEX_SITE_URL`, the production deployment still needs its own.
- If observability matters, configure it at deploy time. Don't treat it as a local-only feature you'll wire "later."

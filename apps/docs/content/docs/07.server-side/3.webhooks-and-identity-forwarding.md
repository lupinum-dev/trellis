Handle webhooks and automation traffic without pretending they're browser sessions. Identity forwarding verifies the request came from a system you trust, then forwards who the real caller is and, optionally, which app [appIdentity](/docs/concepts/glossary#appIdentity) it is acting for — guards and authorization still run like any other call.

If you want one concrete app that shows browser auth, MCP actingFor, and verified webhook traffic all hitting the same protected handler surface, read `examples/07-mcp-reference` in the repo.

```ts [server/api/webhooks/task-create.post.ts]
import { createError } from 'h3'
import { subject } from '@lupinum/trellis/auth'
import { api } from '#trellis/api'
import { delegateToUser, serverConvexMutation } from '#trellis/server'

function getWebhookBotUserId() {
  const userId = process.env.WEBHOOK_BOT_USER_ID
  if (!userId) {
    throw createError({
      statusCode: 500,
      statusMessage: 'WEBHOOK_BOT_USER_ID is not configured.',
    })
  }
  return userId
}

export default defineEventHandler(async (event) => {
  await verifyWebhookSignature(event) // your webhook verification
  const body = await readBody(event)
  const actorUserId = getWebhookBotUserId()

  return await serverConvexMutation(
    event,
    api.tasks.create,
    { projectId: body.projectId, title: body.title },
    {
      auth: 'trusted',
      caller: {
        kind: 'service',
        serviceId: 'task-webhook',
        subject: subject.service('task-webhook'),
      },
      actingFor: await delegateToUser({
        userId: actorUserId,
        allow: true,
        reason: 'verified task webhook',
      }),
    },
  )
})
```

Two checks, one flow: Nitro verifies the request is real, then the Convex handler runs on the trusted path with an explicit server-owned caller identity and optional actingFor target.

## The rule that matters

::caution
Do not treat external server traffic like a browser session. "It's a webhook" is not a permission grant. The webhook is a caller identity; business authorization still applies.
::

A webhook with a valid signature can reach your endpoint. It still has to act _as_ someone — some user, some system bot — and guards still decide what that someone can do. Skipping this step is how teams ship webhook handlers that can do more than any human user.

## What Trellis verifies server-side

Trusted-forwarding payloads carry one signed transport field or Trellis denies
the call:

- `_trellisForwarding` — a short-lived signed envelope binding the Convex
  function ref, args hash, caller subject, caller/actingFor payloads,
  purpose, expiry, replay id, and key id.

That is the only live forwarding transport. Legacy raw forwarding public args
are not supported app authoring surfaces and must not appear in business args.

The signing secret lives in `CONVEX_IDENTITY_FORWARDING_KEY`, set both in your
local env **and** in the Convex deployment handling the traffic. Trellis server
helpers create the envelope for you when you use `auth: 'trusted'`; most app
code should not build envelopes by hand.

In production, Trellis rejects identity-forwarding keys that are obviously weak
or placeholder-shaped. Short values and strings like `replace-me...` are treated
as configuration failures, not as "good enough for now."

Forwarded caller and actingFor payloads live inside the signed envelope.
Treat them as verified transport metadata, not as public request arguments. A
valid envelope authenticates the forwarding boundary; it still does not grant
business permission.

Verification fails closed. Malformed envelopes, unknown key ids, wrong audience
or purpose, wrong function refs, args-hash drift, expired envelopes, and replayed
operation execution tokens are denials, not warnings.

Logs and observations must stay redacted. Do not log raw envelopes, raw keys,
bearer tokens, caller or actingFor payloads, subjects, JTIs, tenant keys, or
confirmation payloads. Use error codes and redacted summaries.

::warning
Missing `CONVEX_IDENTITY_FORWARDING_KEY` on the Convex side is the most common identity-forwarding failure. Browser auth will work fine; trusted traffic will reject. `trellis doctor` can catch missing local key sources, but it cannot verify the deployed Convex environment for you.
::

## Resolve a real app appIdentity — don't fork policy

The right pattern for webhooks isn't "webhook can do anything." It's:

1. Verify the signed identity-forwarding envelope.
2. Forward a real server-owned caller and, when needed, delegate to the user the webhook is acting for.
3. Resolve the app appIdentity from that delegated user and run the normal guards.

That way webhook traffic goes through the same guards, the same permission checks, and the same tenant scoping as browser traffic. There's no second "integration bypass" policy model to maintain, and no synthetic bot-user path to keep in sync.

If your webhook needs different permissions from any human user, model that deliberately in your auth layer. Do not treat a placeholder bot user as the default answer, and do not skip the appIdentity entirely.

## Browser auth vs. identity forwarding auth

They solve different problems:

|                     | Browser auth                   | Identity forwarding auth                               |
| ------------------- | ------------------------------ | ------------------------------------------------------ |
| Answers             | Who is the signed-in end user? | Is this server call allowed to inject a user identity? |
| Token source        | Session cookie → Convex JWT    | `CONVEX_IDENTITY_FORWARDING_KEY`                       |
| Replaces the other? | No                             | No                                                     |

A valid browser cookie doesn't substitute for identity-forwarding verification on a webhook. An identity-forwarding key doesn't substitute for business authorization in the handler.

## `auth: 'none'` is narrower than it sounds

`auth: 'none'` is right for calling a genuinely public handler, or for a route that owns its whole security model. It's wrong as a way to dodge a missing identity-forwarding setup.

If the Convex handler has a guard, and you're calling it with `auth: 'none'`, something's wrong — either the guard shouldn't be there, or the call shouldn't be `none`.

## Common pitfalls

- Calling protected Convex handlers from a webhook without trusted verification — the handler will reject, you'll debug "why is my webhook breaking" for an hour.
- Treating the trusted key as authorization instead of identity forwarding.
- Sending forwarded `caller` or `actingFor` as normal public args.
- Logging raw envelopes, raw keys, bearer tokens, forwarded identity payloads,
  subjects, JTIs, tenant keys, or confirmation payloads.
- Forgetting `CONVEX_IDENTITY_FORWARDING_KEY` on the Convex deployment (only setting it locally).
- Shipping a weak or placeholder identity-forwarding key into production.
- Building a parallel "integration permissions" model instead of resolving a real delegated appIdentity.
- Using `auth: 'none'` to silence a guard you should have configured properly.

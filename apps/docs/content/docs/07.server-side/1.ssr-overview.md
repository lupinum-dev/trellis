Trellis renders on the server with real auth and real data, then hands off to live subscriptions on the client. No protected-page flash, no "logged in?" guess, no separate data system between SSR and live.

```ts
// Same composable. Server-rendered, hydrated, then live.
const { data: tasks } = await useConvexQuery(api.tasks.list, { status: 'active' })
```

This page is the mental model. Once it clicks, the rest of the server-side section (routes, webhooks, identity forwarding) reads as "different callers into the same business layer."

## What SSR actually does

When a request comes in with `auth.enabled: true`, the Nitro server plugin runs three steps before Vue starts rendering:

1. Read the Better Auth session cookie from the request.
2. Exchange it for a Convex JWT.
3. Hydrate token + user state into Nuxt `useState`.

That's why `isAuthenticated` can be `true` on the first paint — the server already did the work, and the client receives it already decided.

## Why protected pages don't flash

Without SSR auth, a protected page has to either:

- render the shell, wait, then redirect (ugly flash), or
- block rendering client-side (breaks SSR), or
- guess from a cookie (unreliable).

With SSR auth, the server already knows which of three states applies:

- authenticated → render the real page;
- unauthenticated → middleware redirects before render;
- misconfigured → dev mode fails loudly (see below).

The client hydrates that decision. There's no "client-side guess" stage to flash through.

## How queries fit in

`useConvexQuery` is SSR-aware by default. One lifecycle, three phases:

1. SSR awaits the first value and renders real data into HTML.
2. The client hydrates with the same state — no refetch.
3. A live subscription takes over. `data` stays reactive.

::note
If you don't want the SSR phase for a given query, pass `server: false`. That's right for non-critical dashboards and activity feeds — see [Queries](/docs/data-fetching/queries#skip-ssr-for-non-critical-data).
::

## Auth readiness vs. appIdentity readiness

This one trips teams up.

SSR can know **who is signed in** (the [caller](/docs/concepts/glossary#caller)) without knowing **who they are in your app** (the [appIdentity](/docs/concepts/glossary#appIdentity)). Those are separate resolutions:

- Auth hydration comes from Better Auth ↔ Convex JWT exchange.
- AppIdentity + [access context](/docs/concepts/glossary#permission-appIdentity) come from your Convex queries.

A valid session can arrive before the `users` row loads, or before `useAccess()` has its context. Don't conflate the two.

::warning
Treating "auth is ready" as "permissions are ready" is how you ship a page that tries to read permissions with a `null` context. Use `useAccess()` state for permission readiness; use `useConvexAuth()` state for auth readiness.
::

## Dev fails loudly on misconfig

In development, SSR auth misconfiguration surfaces as an error, not as a quiet anonymous fallback. That's on purpose — a quiet fallback hides the real bug.

Common triggers:

- Wrong `SITE_URL`.
- Wrong `CONVEX_SITE_URL` (custom HTTP Actions domain).
- Better Auth routes not registered in `convex/http.ts`.
- `BETTER_AUTH_SECRET` mismatch between local env and Convex deployment.

If dev is yelling, run [`trellis doctor`](/docs/auth-security/auth-troubleshooting) — it checks all of these in one pass.

## Optional: SSR auth cache

Token exchange is cheap but not free. Under load you can cache it:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: {
      cache: { enabled: true, ttl: 60 },
    },
  },
})
```

Off by default. Turning it on trades a small correctness window for fewer round-trips per SSR request — right for traffic-heavy deployments, not for local dev.

## SSR vs. server helpers

Two different things, easy to confuse:

- **SSR page rendering** — Nuxt rendering a route for a user browser request.
- **Nitro server routes** — your own `/api/*` handlers calling Convex on the server.

Both run on the server. They're not the same flow. If you're writing a `/api/*` handler, go to [Server routes](/docs/server-side/server-routes).

## Common pitfalls

- Using `useConvexAuth()` state as a stand-in for permission readiness.
- Chasing a "protected flash" that isn't there — if it's flashing, something's wrong with the SSR auth path, not the UI.
- Enabling the auth cache in dev to "fix" slow sign-in (slow sign-in is usually misconfig — fix the cause).
- Treating server helpers and SSR rendering as the same flow.

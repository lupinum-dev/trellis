::caution
This page is for authors of reusable Trellis-aware packages. **Most app teams can ignore it.** Normal app features should use root Convex handlers, server helpers, operations, and MCP projection.
::

## Who this is for

Use the bridge only when you are building a packaged integration that another Trellis app installs, such as a CMS or billing component with its own Convex component internals.

Do not use it for ordinary app features. If the code belongs to the app, keep one app-owned backend model:

- root `query.public(...)`, `query.protected(...)`, `mutation.protected(...)`, or operation handlers;
- thin Nitro routes that call those handlers through server helpers;
- MCP tools that project root handlers or operations.

The bridge exists for package boundaries, not for avoiding app wiring.
It is the stable automation seam for packaged Trellis component integrations,
not a day-to-day app architecture pattern.

## Ownership model

A bridge-aware package has four owners:

| Owner                     | Owns                                                                                           |
| ------------------------- | ---------------------------------------------------------------------------------------------- |
| Packaged integration      | The bridge manifest, package docs, and generated host file contract.                           |
| Host Trellis app          | Root app policy, appIdentity resolution, tenant model, generated Convex refs, and env vars.    |
| `@lupinum/trellis-bridge` | Manifest loading, file rendering, managed edits, install checks, and drift checks.             |
| Product package           | End-user setup docs. Integration users should read product docs, not Trellis bridge internals. |

This boundary keeps bridge machinery out of the beginner Trellis path. Trellis can pressure-test the bridge with integration-shaped fixtures without making product setup part of the Trellis starter story.

## Package boundary

Bridge package-author APIs live in `@lupinum/trellis-bridge/manifest`.

```ts [src/module/bridge-manifest.ts]
import { defineComponentBridgeManifest } from '@lupinum/trellis-bridge/manifest'
```

Generated or host-side bridge modules may use the component runtime subpath:

```ts [convex/acmeCms/public.ts]
import { createComponentBridge } from '@lupinum/trellis-bridge/component'
```

Do not import bridge helpers from `@lupinum/trellis`, root barrels, backend builders, or normal app modules. If regular app code needs to call Convex, use the server helpers and root handlers instead.

## What the manifest declares

A packaged integration exports a default bridge manifest at `./convex/manifest`. The manifest declares the exact host files the package needs Trellis to install or check.

```ts [src/module/bridge-manifest.ts]
import { defineComponentBridgeManifest } from '@lupinum/trellis-bridge/manifest'

export default defineComponentBridgeManifest({
  packageName: '@acme/cms',
  version: '0.1.1',
  renderFiles: [
    {
      relativePath: 'convex/acmeCms/public.ts',
      content: [
        "import { createAcmeCmsBridge } from '@acme/cms/convex'",
        "import { query, mutation } from '../_generated/server'",
        "import { components } from '../_generated/api'",
        '',
        'const bridge = createAcmeCmsBridge({',
        '  query,',
        '  mutation,',
        '  cms: components.acmeCms,',
        '})',
        '',
        'export const listEntries = bridge.listEntries',
        'export const publishEntry = bridge.publishEntry',
        '',
      ].join('\n'),
    },
  ],
  managedEdits: [
    {
      relativePath: 'convex/convex.config.ts',
      apply(current) {
        return (
          current ?? "import { defineApp } from 'convex/server'\n\nexport default defineApp()\n"
        )
      },
    },
  ],
})
```

Publish the manifest through `package.json`:

```json [package.json]
{
  "exports": {
    "./convex/manifest": {
      "types": "./convex/manifest.d.ts",
      "import": "./convex/manifest.js"
    }
  }
}
```

The manifest is package infrastructure. It is not an app feature registry, not a public API promise for end users, and not a general-purpose code generator.

## Install and drift workflow

Host apps install and check bridge files through product-owned tooling. For example, a CMS package should expose its own `init`, `bridge check`, or `doctor` command rather than making end users learn Trellis bridge internals.

Generated files are explicit host files. The host app can review them, commit them, and run normal Convex codegen against them.

If a package's Nuxt module needs startup validation, call the Node-only drift helpers from module or setup code, not from Convex functions:

```ts
import { checkBridgeDrift, loadManifestFromPackage } from '@lupinum/trellis-bridge'

const manifest = await loadManifestFromPackage('@acme/cms', rootDir)
const drift = await checkBridgeDrift(manifest, rootDir)
if (drift.length > 0) {
  throw new Error('@acme/cms bridge files are missing or stale.')
}
```

Keep the product-facing command product-owned. For example, a CMS package can wrap this in its own `init` or `doctor` command so CMS users do not need to learn Trellis bridge internals.

## Forwarding and authorization

Bridge runtime helpers sign `_trellisForwarding` envelopes when a host bridge function calls into a component ref. The envelope authenticates the transport boundary and binds purpose, function ref, args hash, subject, expiry, and key id.

Anonymous public bridge calls are intentionally unsigned and pass only the declared business args. Signed bridge forwarding is for non-anonymous principals with canonical subjects.

Component-side verification reads the identity forwarding key from component-side configuration, usually `CONVEX_IDENTITY_FORWARDING_KEY`, or an explicit bridge key option. Do not transport signing keys in Convex args.

That still does not grant business permission. Component-side handlers must resolve caller and appIdentity, run guards, load records, bind tenants, authorize, and execute exactly like any other protected handler.

The package author must not expose identity-shaped public args such as `caller`, `actingFor`, or raw forwarding keys. Identity travels inside the signed forwarding envelope.

## What not to do

- Do not use bridge helpers in normal app feature code.
- Do not call internal component refs directly after generating root bridge files.
- Do not build a separate package-specific authorization model in the bridge layer.
- Do not ask end users to edit generated bridge files by hand.
- Do not treat generated `convex/manifest.js` as source. Keep an authored TypeScript source and emit the package-facing manifest during build.
- Do not make CMS setup a Trellis starter unless the product owner deliberately chooses that surface. Product setup belongs to the product package.

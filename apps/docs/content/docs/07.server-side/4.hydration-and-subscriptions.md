How the server-rendered data becomes a live subscription on the client — without a refetch, without a flash, and with one subscription shared across every component that reads the same query. This page is the mechanics. The user-facing story is in [SSR overview](/docs/server-side/ssr-overview); the UI signals are in [Connection state](/docs/data-fetching/connection-state).

## The handoff, step by step

For a normal `useConvexQuery`:

1. SSR awaits the first result and renders it into HTML.
2. The HTML ships to the client with the data inlined as Nuxt state.
3. The client hydrates — same data, no refetch.
4. The Convex client takes over. A live subscription attaches.
5. `data` updates reactively from there.

::note
The key property: step 3 doesn't trigger a new network call. If you see a fetch on page load in DevTools that duplicates the server-rendered data, something's wrong with hydration — usually a shape mismatch between what SSR stored and what the client tried to read.
::

## One subscription, many consumers

Trellis dedupes live subscriptions across the whole Nuxt app. If three components call `useConvexQuery(api.tasks.list, {})` with the same args, there's one websocket subscription — not three.

| Behavior            | What it means                                         |
| ------------------- | ----------------------------------------------------- |
| Shared subscription | Multiple consumers observe the same state             |
| Reference-counted   | Subscription released when the last consumer unmounts |
| Args-keyed          | Different args = different subscription (correct)     |

This matters for pages with a sidebar and a main column both reading the same list. You don't pay 2× the traffic for showing the same data twice.

## Hydration isn't a disconnect

The first moment after SSR is special. The websocket hasn't connected yet. Browsers in that state _look_ offline if you check `isConnected`. They aren't — they just haven't had time to connect.

Trellis suppresses offline UI during that grace window. Use `shouldShowOfflineUi` from `useConvexConnectionState`, not `isConnected === false`, or you'll ship a banner that flashes on every page load. See [Connection state](/docs/data-fetching/connection-state).

## Auth hydration and query hydration are separate

Near each other, easy to confuse:

- **Auth hydration** — Better Auth session cookie → Convex JWT → Nuxt state. One handoff.
- **Query hydration** — SSR-fetched data → Nuxt state → live subscription. A different handoff, per query.

They can be in different readiness states. Auth can be hydrated while a specific live subscription hasn't connected yet. A query can have SSR data while access context is still loading. Don't check one to reason about the other.

## First connect vs. reconnect

The Trellis connection runtime distinguishes three phases, and the UI should too:

| Phase          | When it happens                         | UI usually shows       |
| -------------- | --------------------------------------- | ---------------------- |
| `connecting`   | Initial websocket setup after hydration | Nothing (grace window) |
| `connected`    | Steady state                            | Nothing                |
| `reconnecting` | Was connected, isn't now                | Banner/indicator       |

A reconnect after a real disconnect is worth telling the user about. The initial connect after SSR is not — they just got here.

## When to investigate the handoff

These are symptoms of handoff problems, not plain data bugs:

- SSR data renders correctly, but the page never receives live updates.
- Multiple components fire duplicate live traffic for the same query.
- Offline banner flashes on page load.
- Auth is ready but permission-gated UI stays unset.

If a query is returning the _wrong value_, that's not a handoff issue — that's a query or args issue. Handoff problems are about state readiness, not state content.

## Common pitfalls

- Using `!isConnected` as an offline signal (flashes on every page load).
- Treating auth hydration as a proxy for query or permission readiness.
- Assuming duplicate traffic for the same query means broken dedupe — check if the args are actually the same (different refs, different object identity).
- Looking for handoff bugs when the real bug is a wrong query result.

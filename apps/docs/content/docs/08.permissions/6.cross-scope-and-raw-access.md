Trellis gives you one default path and two explicit escape hatches. They are not interchangeable. Use the narrowest one that does the job and make the reason visible in review.

| Surface                                      | Bypasses               | Still enforces                         | Requires        | Emits                      |
| -------------------------------------------- | ---------------------- | -------------------------------------- | --------------- | -------------------------- |
| `ctx.db`                                     | Nothing                | Isolation, services, triggers          | —               | —                          |
| `ctx.db.escapeIsolation({ reason })`         | Isolation              | Service rules, triggers, runtime hooks | Inline `reason` | `db.escape_isolation.used` |
| `query.unsafe(...)` / `mutation.unsafe(...)` | Protected handler path | Convex runtime + surrounding infra     | Typed `permit`  | `unsafe.handler.used`      |

Preference order, always: `ctx.db` → `ctx.db.escapeIsolation(...)` → unsafe lanes. Do not skip rungs.

## Default: `ctx.db`

For tenant-isolated tables, `ctx.db` enforces the active tenant boundary on both reads and writes. Use it unless you have a concrete reason not to. Most handlers end up here once the appIdentity model is right.

## Narrow escape hatch: `ctx.db.escapeIsolation({ reason })`

Use this when the handler is still the right protected shape, but the data access intentionally crosses tenant boundaries.

```ts [convex/agency/posts.ts]
export const listAllPosts = query.protected({
  args: {},
  guard: authed,
  handler: async (ctx) => {
    return await ctx.db
      .escapeIsolation({
        reason: 'Agency portfolio views span multiple client workspaces.',
      })
      .query('posts')
      .collect()
  },
})
```

What still applies:

- Service rules.
- Triggers and runtime hooks.
- Observability — every call emits `db.escape_isolation.used` with the attached `reason`.

Right for: admin, operator, onboarding, and portfolio flows that genuinely need to cross-scopes while keeping the rest of the Trellis runtime intact.

## Sharp escape hatch: unsafe lanes

`query.unsafe(...)` / `mutation.unsafe(...)` are for cases where the handler itself must be declared outside the normal protected builder path.

```ts [convex/features/uploads/domain.ts]
import { unsafe } from '@lupinum/trellis/backend'

export const generateUploadUrl = mutation.unsafe({
  permit: unsafe.permit({
    kind: 'preTenantUpload',
    reason: 'Generate upload URLs before a concrete tenant-scoped record exists.',
    scope: ['assets'],
  }),
  args: {},
  handler: async (ctx) => {
    requireAuth(await ctx.appIdentity())
    return await ctx.storage.generateUploadUrl()
  },
})
```

::caution
`query.unsafe(...)` / `mutation.unsafe(...)` are not nicer `escapeIsolation(...)` variants. If the protected handler shape still works and you only need cross-scope DB access, stay on `query.protected(...)` / `mutation.protected(...)` and use `ctx.db.escapeIsolation({ reason })` inside the handler.
::

## How to choose

::steps{level="3"}

### Start with `ctx.db`

If the appIdentity model is right and the table is tenant-isolated, this covers almost everything.

### Step up to `escapeIsolation(...)` when the action truly crosses tenants

Onboarding, workspace switching, agency dashboards. Ask: _would I be okay if this ran against the active tenant only?_ If yes, stay on `ctx.db`.

### Use unsafe lanes only when the handler itself must be declared outside the normal protected path

Almost never the first answer. If you reach for one, `permit: unsafe.permit(...)` is mandatory and the handler emits `unsafe.handler.used` at runtime.

::

## Real-app cases

From the multi-workspace example:

- Current-workspace work → normal tenant-scoped handlers.
- Onboarding and workspace switching → explicit tenant escape flows.
- Agency dashboards reading membership across workspaces → tenant-escape reads on purpose.
- Upload URL generation before a concrete record exists → `mutation.unsafe(...)`.

Each one keeps the exception visible. No quiet handler that silently crosses boundaries.

## Leave the reason at the call site

```ts
const db = ctx.db.escapeIsolation({
  reason: 'Onboarding creates a tenant before the caller has a tenant-bound appIdentity.',
})

return await db.insert('workspaces', { ... })
```

This is not style. Trust-boundary changes need to be obvious in review. If the reviewer has to infer why the code crossed tenants or used an unsafe lane, the reason belongs closer to the call site.

## Common pitfalls

- Treating `ctx.db.escapeIsolation(...)` as normal app data access. Normal access is `ctx.db`.
- Reaching for an unsafe lane when `ctx.db.escapeIsolation({ reason })` would preserve the guarantees you want.
- Hiding an escape hatch inside a helper where the boundary is no longer obvious to reviewers.
- Using an unsafe lane to avoid modeling the right appIdentity or tenant flow. That becomes a maintenance problem immediately.

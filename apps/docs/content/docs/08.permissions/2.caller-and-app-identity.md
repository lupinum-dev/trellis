This page matters once the app has more than one meaningful caller shape: browser plus server, browser plus MCP, or browser plus workspace policy.

If you are still building the first signed-in todo app, you can skip this for now and come back later.

A [caller](/docs/concepts/glossary#caller) says how the call arrived. An [appIdentity](/docs/concepts/glossary#appIdentity) says who that is in your app. Collapsing them is how policy bugs happen, so Trellis keeps them strictly separate.

| Layer       | Shape                                            | Decides                   |
| ----------- | ------------------------------------------------ | ------------------------- |
| Caller      | Transport kind (anonymous, user, agent, service) | How to resolve the caller |
| AppIdentity | App-owned identity (role, tenant, membership)    | What the caller can do    |

Same person opening your app in a browser and asking your agent to do the same thing resolves to the **same appIdentity**. The transport changed; the business identity didn't. That's the point.

## Caller: transport identity

From a workspace/MCP app:

```ts [convex/auth/caller.ts]
export type TeamTodoPrincipal =
  | { kind: 'anonymous'; subject: 'system:anonymous' }
  | { kind: 'user'; authKey: string; subject: `auth:${string}` }
  | { kind: 'agent'; agentId: string; subject: `agent:${string}`; provider: 'mcp' }
  | { kind: 'service'; serviceId: string; subject: `service:${string}` }
```

A caller tells you:

- Who the transport says they are (browser session, forwarded agent call, nothing).
- Sometimes a thin metadata hint (`provider: 'mcp'`).

What it explicitly doesn't decide: what the caller is allowed to do. That's the appIdentity's job.

## AppIdentity: app-owned business identity

```ts [convex/auth/appIdentity.ts]
export type AppIdentity = DefaultAppIdentity & {
  role: Role
  workspaceId?: Id<'workspaces'>
}
```

This is what guards and authorization actually read. Role, tenant membership, any app-specific scope.

## The resolution flow

Four steps in this order, every request:

::steps{level="3"}

### Resolve the caller

The runtime fills this in from the request.

### Resolve the appIdentity

Your `resolve(...)` turns that caller into an app identity (or returns `null` if there isn't one).

Missing appIdentity wiring is different from a resolved `null` appIdentity. Missing wiring means the runtime or test context was not set up correctly, so protected paths fail closed. A resolved `null` means the resolver ran and found no app appIdentity for this caller; protected guards deny it like any other unauthorized caller.

### Run guards and authorization

Against the appIdentity, not the caller.

### Execute the handler

If the checks passed.

::

In an MCP/server path, represented-user delegation comes from the signed
`actingFor` envelope. An agent caller does not become a user just because it has
an agent id:

```ts [convex/auth/appIdentity.ts]
import { getSubjectValue } from '@lupinum/trellis/auth'
import type { ActingFor } from '@lupinum/trellis/backend'

export async function getAppIdentityFromCaller(
  ctx,
  _args,
  caller,
  actingFor: ActingFor | null,
): Promise<AppIdentity | null> {
  const delegatedUserId = getSubjectValue(actingFor?.subject, 'user')
  if (delegatedUserId) {
    return await loadActorByUserId(ctx, delegatedUserId)
  }

  switch (caller.kind) {
    case 'anonymous':
    case 'agent':
    case 'service':
      return null
    case 'user':
      return await loadActorByAuthKey(ctx, caller.authKey)
  }
}
```

Notice what this doesn't do:

- It doesn't use `caller.kind` as an authorization signal.
- It doesn't invent a separate role system for agent callers.
- It doesn't let MCP traffic drift onto a parallel policy track.

## Why not put role on the caller

Because then business authorization would be coupled to how the call arrived. Moving an action to a new transport (CLI, webhook, agent) would mean duplicating your role model.

Trellis keeps transport metadata on the caller (like `provider: 'mcp'`) and business authority on the appIdentity. If you need a forwarded caller with real trust, model it deliberately — don't smuggle role claims through transport shape.

::note
"Browser, server, webhook, and agent callers can share one backend model." They still don't replace Convex business authorization. That's the shape Trellis wants.
::

## Permission appIdentity vs. strict appIdentity

Most apps need two slightly different appIdentity views:

| View                | Used for                                         | Accepts appIdentity without `workspaceId`? |
| ------------------- | ------------------------------------------------ | ------------------------------------------ |
| `getAccessIdentity` | Frontend access context, onboarding flows        | Yes                                        |
| `getAppIdentity`    | Tenant-scoped mutations that require a workspace | No                                         |

The workspace scaffold splits them explicitly. It keeps onboarding and workspace-required mutations from fighting each other: you can't open the app without a tenant, but the second you act inside one, the appIdentity must have `workspaceId`.

## Common pitfalls

- Using `caller.kind` as the authorization decision. It's not — it's metadata about the transport.
- Encoding app roles into every transport caller. The appIdentity layer owns role.
- Reusing one strict appIdentity resolver everywhere, then wondering why onboarding users disappear.
- Treating a missing `appIdentity()` accessor as the same thing as a resolver that returned `null`.
- Letting browser and MCP calls drift onto separate authorization systems.

Let Trellis own the tenant boundary. Configure one thing in `defineTrellis(...)` and `ctx.db` reads and writes stop being able to leak across-scopes — a handler that forgets a manual tenant filter can't quietly serve the wrong workspace's data.

This is infrastructure, not your app's main authorization model. Trellis uses runtime row enforcement to keep tenant/service boundaries intact, while business authorization stays in handler code.

```ts [convex/trellis.ts]
export const { mutation, query, unsafe } = defineTrellis(
  { query: generatedQuery, mutation: generatedMutation },
  {
    appIdentity: getAppIdentityFromCaller,
    isolation: {
      tables: ['todos'],
    },
  },
)
```

That's the whole contract. `ctx.db` on `todos` now scopes to the active appIdentity's tenant.

## Configure it in `defineTrellis(...)`

Default tenant field is `workspaceId`. If your tables use a different name, set it explicitly:

```ts
isolation: {
  tables: ['posts'],
  field: 'organizationId',
}
```

Trellis validates the shape at setup:

- The table list can't be empty.
- The table list can't contain duplicates.
- The field can't be blank.
- Every listed table must carry the configured field _and_ expose an index on it.

If any of those don't hold, module validation fails loudly instead of producing a runtime that appears to work.

## Only list tables that actually have tenant scope

Easy mistake: listing every table. Don't.

List only tables that:

- Store the configured tenant field on every row.
- Have an index on that field for efficient scoping.
- Should be filtered by the active appIdentity's tenant.

::warning
Listing a table here that doesn't have the tenant field or index will fail validation. Trellis needs the index to scope reads efficiently — a missing index would turn every scoped query into a table scan.
::

## What default `ctx.db` does

Once a table participates in isolation:

| Operation                                | Behavior                                                  |
| ---------------------------------------- | --------------------------------------------------------- |
| Read                                     | Scoped to the active appIdentity's tenant                 |
| Write                                    | Must include the matching tenant field                    |
| `ctx.db.get(id)` on another tenant's row | Throws `Document belongs to a different isolation scope.` |
| List queries                             | Only see rows for the current tenant                      |

::note
A direct cross-scope `ctx.db.get(...)` doesn't quietly return `null`; it raises. That's the safer default — handlers that forget a manual tenant check don't get to leak by accident.
::

## Isolation ≠ business authorization

Two different questions:

| Isolation asks                              | Business authorization asks               |
| ------------------------------------------- | ----------------------------------------- |
| Does this row belong to the current tenant? | Can this appIdentity archive a project?   |
| —                                           | Can this member edit someone else's task? |
| —                                           | Can this user issue MCP keys?             |

You still need guards and authorization for business policy inside the tenant boundary. Isolation is the floor, not the whole building.

Do not try to move app authorization into the isolation layer. Use it to prevent cross-scope leaks, then express business rules in `guard` / `load` / `authorize` / `handler`.

## When not to use it

Don't force bootstrap flows through isolation-scoped handlers. Workspace creation, workspace switching, admin portfolio views — these legitimately need to cross the tenant boundary. Trellis exposes explicit escape hatches for exactly those cases; see [Tenant escape and unsafe access](/docs/permissions/cross-scope-and-raw-access).

## Common pitfalls

- Listing tables that don't store the tenant field. Validation will catch it; fix the schema first.
- Assuming isolation replaces business authorization. It doesn't — it's the outer layer.
- Using one tenant field name in code and another in schema/indexes. They must match.
- Forcing bootstrap flows through isolation-scoped handlers "to stay pure." Use a cross-scope handler deliberately.

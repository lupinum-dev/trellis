Read this page when one protected handler has turned into several and you need named, reusable backend checks.

If the app still has only one small signed-in rule, the getting-started guide is enough for now.

Name your checks. Compose them. Reuse them across handlers. [Guards](/docs/concepts/glossary#guard) turn "is this allowed?" into readable policy instead of anonymous booleans spread across your codebase.

```ts [convex/auth/guards.ts]
export const hasWorkspace = defineGuard<AppIdentity>(
  'Workspace member',
  (appIdentity) => !!appIdentity?.workspaceId,
)

export const canCreateTodo = defineGuard<AppIdentity>(
  'Create todo',
  hasWorkspace.and(hasRole('owner', 'admin', 'member')),
)
```

That reads like policy, and it stays one check — not five scattered `if` statements.

## Define static guards

The primitive is `defineGuard(label, check)`:

```ts [convex/auth/guards.ts]
export const hasWorkspace = defineGuard<AppIdentity>(
  'Workspace member',
  (appIdentity) => !!appIdentity?.workspaceId,
)

export const hasRole = (...roles: Doc<'users'>['role'][]) =>
  defineGuard<AppIdentity>(
    `role:${roles.join('|')}`,
    (appIdentity) => !!appIdentity && roles.includes(appIdentity.role),
  )
```

Named and reusable. The label shows up in errors and observability — a guard called `'Workspace member'` tells you more than an inline `!!appIdentity?.workspaceId` ever will.

## Compose guards

Guards support `.and(...)`, `.or(...)`, and `.not()`:

```ts
export const canCreateTodo = defineGuard<AppIdentity>(
  'Create todo',
  hasWorkspace.and(hasRole('owner', 'admin', 'member')),
)
```

Two wins from composition:

- The code reads like policy, not control flow.
- The structure stays inspectable — Trellis can walk the tree for diagnostics and observability.

## Resource-bound guards: use a factory

When a check needs loaded data, wrap it in a factory:

```ts [convex/auth/guards.ts]
export const isOwnerOf = (resource: { ownerId: string }) =>
  defineGuard<AppIdentity>(
    `owner:${resource.ownerId}`,
    (appIdentity) => !!appIdentity && appIdentity.userId === resource.ownerId,
  )

export const canUpdateTodo = (todo: { ownerId: string }) =>
  defineGuard(
    'Update todo',
    hasWorkspace.and(hasRole('owner', 'admin').or(hasRole('member').and(isOwnerOf(todo)))),
  )
```

The static rule vocabulary (`hasWorkspace`, `hasRole`) stays separate from the data-bound rule (`isOwnerOf`). You get reuse on the static parts and clarity on the per-resource parts.

## Static permissions are a separate layer

Use `definePermission(...)` for named, projected recordAccess that the frontend or MCP should read:

```ts [convex/auth/permissions.ts]
export const todoCreate = definePermission({
  key: 'todo.create',
  label: 'Create todo',
  roles: ['owner', 'admin', 'member'],
  check: hasWorkspace.and(hasRole('owner', 'admin', 'member')),
})
```

That is intentionally narrower than a general authorization DSL. Static permissions project to `useAccess().can(todoCreate)`. Record-bound checks like `canUpdateTodo(todo)` stay plain guard factories.

## `authRequired` is a signed-in boundary, not a normal guard

`authRequired` is special — it can't be composed with `.and(...)`, `.or(...)`, or `.not()`. It requires a non-anonymous caller and a resolved appIdentity row, but it does not enforce a business recordAccess by itself. Use it when the handler needs "signed-in user with a real appIdentity" before the rest of the workflow runs.

```ts [convex/workspaces/create.ts]
export const createWorkspace = mutation.protected({
  guard: authRequired,
  args: { name: v.string(), slug: v.string() },
  handler: async (ctx, args) => {
    const caller = await ctx.caller()
    // ...
  },
})
```

For everything else, compose real predicate guards that encode the business rule.

## Keep guards coarse

The Trellis pattern is three phases:

1. Guard at the outer boundary.
2. Load the resource.
3. Authorize against the loaded resource.

Don't try to fold every ownership check into one giant top-level guard. When the decision needs data, move it to [authorize](/docs/permissions/authorization-and-can) after load. Guards own the coarse "should this call even proceed?" question.

## Guards and frontend permissions are projections of each other

Same policy model, different surface:

- Backend handlers consume guards directly.
- Frontend pages consume a projection through `defineAccessContext(...)` and `useAccess()`.

The browser gets a _projection_ of backend policy — never the policy engine itself. That's why `useAccess().can(permission)` doesn't re-run guards in the browser; it reads what the server decided.

## Common pitfalls

- Writing anonymous inline booleans everywhere instead of named guards. The labels pay for themselves in errors.
- Treating `authRequired` like a composable guard. It isn't.
- Encoding transport checks (e.g. "is this an MCP call?") as business authorization. Transport goes on the caller; role goes on the appIdentity.
- Stuffing tenant bootstrap, role checks, and resource ownership into one unreadable predicate — split them across guard, load, and authorize.

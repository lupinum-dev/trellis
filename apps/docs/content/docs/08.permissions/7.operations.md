Read this page when the same business action has to exist in more than one place: confirmation UI, admin surface, server workflow, or MCP tool.

If a plain protected mutation still explains the whole feature cleanly, stay
with that. Operations are for the point where reuse, preview semantics, or
cross-surface identity matter.

Some business actions live in multiple places — a button in the UI, a preview query for confirmation UX, an MCP tool for the agent. Write that action once with `defineOperation(...)` and let Trellis project it into each surface. One guard, one load, one authorize, one preview, one handler.

```ts [convex/features/todos/domain.ts]
export const removeTodoOp = defineOperation({
  id: 'todos.remove',
  name: 'removeTodo',
  kind: 'destructive',
  identityForwardingFunctionRef: 'features/todos/domain:remove',
  args: deleteTodo.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canDeleteTodo(todo),
  },
  preview: async (_ctx, _args, { todo }) =>
    operationPreview({
      summary: `Will permanently delete "${todo.title}"`,
      warnings: [
        operationIssue({
          code: 'permanent-delete',
          message: 'This cannot be undone.',
        }),
      ],
      effects: [
        operationEffect({
          kind: 'delete',
          summary: 'Delete todo',
          target: 'todo',
          count: 1,
        }),
      ],
      confirm: {
        operation: 'todos.remove',
        targetId: todo._id,
        affectedCounts: { todos: 1 },
      },
    }),
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id)
    return null
  },
})

export const remove = mutation.protected(removeTodoOp)
export const previewRemove = mutation.protected(previewOf(removeTodoOp))
```

Preview and execute share the same operation definition. The source-of-truth
chain is:

1. operation descriptor or operation definition;
2. protected preview and execute projections;
3. UI, server, MCP, or bridge bindings that point at those projections.

If a projection claims to represent an operation but its operation identity,
kind, args, permission key, preview ref, or execute ref drifts, Trellis should
reject it during registration, doctor, or tests.

## When to reach for an operation

| Good fit                                          | Skip it                                          |
| ------------------------------------------------- | ------------------------------------------------ |
| Destructive action (delete, archive, bulk remove) | Simple non-destructive one-off mutation          |
| Same action needs preview and execute             | No preview needed — a regular handler is clearer |
| Same business rule backs both UI and agent flows  | Single-transport, single-call action             |
| Action deserves a stable contract across surfaces | Tiny local mutation                              |

Operations buy reuse with shared policy. For a trivial mutation, a normal
protected handler is shorter and clearer.

## Register it as execute and preview mutations

```ts
export const remove = mutation.protected(removeTodoOp)
export const previewRemove = mutation.protected(previewOf(removeTodoOp))
```

What you get:

- One execute path, from the operation's `handler`.
- One preview path, projected from the operation's `preview`.
- Shared `guard`, `load`, `authorize` — no second implementation to keep in sync.

The preview isn't a parallel implementation. It's the _preview phase_ of the
same operation, surfaced as its own preview projection.

For destructive operations, preview is side-effect-free. It may read current
state, run the same load and authorization path, compute the neutral preview
envelope, and return confirmation material. It must not mutate app data, call
external services, enqueue jobs, send notifications, or perform irreversible
work.

Execute is not allowed to trust an old preview as authorization. It re-runs the
operation's guard, load, authorize, tenant binding, and drift checks before the
handler applies the mutation. A successful preview means "this is what was
shown"; it is not a grant to skip backend policy.

## Destructive operations have extra requirements

Trellis is strict here on purpose:

::warning
For `kind: 'destructive'` operations, these must be present:

- `id` (stable string — used to bind preview to execute and to identify the tool in MCP).
- `identityForwardingFunctionRef` for the execute projection when preview tokens are issued.
- `preview(...)` — destructive actions must be previewable.
- `destructiveOperations` configured in `defineTrellis(...)` before the operation registers.
- Confirmation and audit tables exist in `convex/schema.ts`: the confirmation
  table needs `tokenHash`, `jti`, `operationId`, `executePath`, `previewPath`,
  `callerKey`, `scopeKey`, `argsHash`, `previewHash`, `createdAt`, `expiresAt`,
  plus `by_token_hash`, `by_jti`, and `by_expires_at` indexes; the audit table
  needs `operationId`, `jti`, `callerKey`, `scopeKey`, `argsHash`,
  `previewHash`, `executedAt`, and `executePath`.
- A confirmation payload returned from preview and required for the strict
  confirmation path.

Skip any of these and Trellis rejects the registration. That's the point:
confirmation binds to previewed state, not to a loose client convention.
::

See [Destructive operations](/docs/mutations/destructive-operations) for the browser-side preview + confirm UX.

## Preview returns one envelope

```ts
preview: async (_ctx, _args, { todo }) =>
  operationPreview({
    summary: 'Delete one todo',
    blockers: [],
    warnings: [],
    effects: [
      operationEffect({
        kind: 'delete',
        summary: 'Delete todo',
        target: 'todo',
        count: 1,
      }),
    ],
    confirm: { operation: 'todos.remove', targetId: todo._id },
    details: { title: todo.title },
  })
```

| Part       | Purpose                                                |
| ---------- | ------------------------------------------------------ |
| `allowed`  | Whether execution may proceed after confirmation       |
| `summary`  | Short human-readable description of the action         |
| `blockers` | Reasons execution is not allowed                       |
| `warnings` | Allowed-but-important risks to surface                 |
| `effects`  | Domain-neutral list of what will change                |
| `confirm`  | Durable confirmation contract bound to execution       |
| `details`  | Optional app-owned structured context for UI or agents |

The envelope is produced by the preview phase, against the same loaded data,
with the same authorization. The `confirm` value is data for later machine
checks; it is not permission by itself.

## Operations are the bridge to MCP tools

`tool.operation(...)` projects an operation into a destructive agent tool. It requires:

- An operation with an `id`.
- An execute ref from that operation.
- When preview is used, a preview ref from the _same_ operation.

If those refs don't match, Trellis rejects the binding. That's how you prevent a
destructive MCP tool from drifting onto a different business action than it
claims. See [MCP: destructive tools](/docs/mcp-tools/destructive-tools).

## Common pitfalls

- Treating preview as optional for destructive actions. It isn't — Trellis requires it.
- Putting side effects in preview. Preview describes work; execute does work.
- Treating preview success as authorization for a later execute call.
- Forgetting `id` on an operation that needs stable projection. You'll hit a registration error.
- Duplicating preview and execute policy in separate handlers. That's the problem operations solve.
- Wrapping tiny local mutations in `defineOperation(...)` because the primitive exists. A regular mutation is fine.

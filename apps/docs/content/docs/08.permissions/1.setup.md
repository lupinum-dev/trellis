Read this page when the app has crossed from "signed-in" into "recordAccess and workspace policy."

If you are still on the public baseline or the first personal auth baseline, skip this page for now. The first protected app does not need a access context query yet.

If you need the handler lane model first, read
[Backend builders](/docs/permissions/backend-builders). That page explains the
public/protected/unsafe split this guide builds on.

Put the policy on the backend once. Project it to the UI with one query and one composable. You don't recompute permissions in the browser; you read what the server decided.

Trellis has one public authorization story for apps:

- Runtime guardrails on `ctx.db` for tenant boundaries.
- Explicit business authorization in handler code.

Do not model app authorization as DB policy. Keep the business rule where the business action lives.

```ts [convex/permissions/context.ts]
import { todoCreate, todoRead } from '../auth/permissions'

export const getAccessContext = query.protected(
  defineAccessContext({
    resolve: getAppIdentity,
    permissions: [todoRead, todoCreate],
  }),
)
```

```vue [app/pages/index.vue]
<script setup lang="ts">
const { can, ready } = useAccess()
</script>
```

One source of truth. The UI is a projection of it.

## The shape of a permissions setup

Five steps, each in a different layer:

::steps{level="3"}

### Resolve an [appIdentity](/docs/concepts/glossary#appIdentity) in Convex

From the caller (who called) to the app identity (who they are in your business model). See [Caller and appIdentity](/docs/permissions/caller-and-app-identity).

### Declare static permissions in app-owned backend code

Named handles for projected UI and MCP recordAccess. Record-bound checks stay in `checks.ts`. See [Guards](/docs/permissions/guards).

### Expose one permission-context query

`defineAccessContext(...)` returns a public query. The browser calls it; the server decides.

### Point Nuxt at that query

One line in `nuxt.config.ts`.

### Read it with `useAccess()`

`can(permission)` and `ready` in your components. That's it.

::

## Start from the scaffold

Fastest way to land on the shipped conventions:

```bash [terminal]
pnpm dlx @lupinum/trellis init my-app --template workspace
```

You get the full Trellis shape:

- `convex/auth/caller.ts`
- `convex/auth/appIdentity.ts`
- `convex/auth/guards.ts`
- `convex/auth/permissions.ts`
- `convex/functions.ts`
- `convex/permissions/context.ts`
- a `getAccessContext` query

You can hand-write the same setup, but the workspace starter is the paved path.

## Define a access context query

The backend query is the source of truth for the frontend auth context:

```ts [convex/permissions/context.ts]
import { teamWorkspacePermissions } from '../auth/permissions'

export const getAccessContext = query.protected(
  defineAccessContext({
    resolve: getAppIdentity,
    permissions: teamWorkspacePermissions,
    extend: async (ctx, appIdentity) => {
      const user = await ctx.db.get(appIdentity.userId)

      return {
        email: user?.email ?? null,
        displayName: user?.displayName ?? null,
      }
    },
  }),
)
```

| Field         | What it does                                                     |
| ------------- | ---------------------------------------------------------------- |
| `resolve`     | Turns the caller into your app appIdentity                       |
| `permissions` | Lists the named static permissions that project into `can`       |
| `extend`      | Adds app-owned fields to the context (plan, usage, display name) |

The base context always includes `userId`, `workspaceId`, `role`, and `can`. You don't wire those by hand.

## Point Nuxt at that query

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: { enabled: true },
    permissions: {
      query: 'permissions/context.getAccessContext',
    },
  },
})
```

If the path is wrong, Trellis fails loudly at setup and `useAccess()` never goes ready — intentional, so a silent miss can't hide the real bug.

## Read permissions in the UI

```vue
<script setup lang="ts">
const { can, ready, role, workspaceId, ctx } = useAccess()

const canCreateTodo = can(todoCreate)
</script>
```

The composable is fail-closed by design:

- Before the access context arrives, `ready` is `false`.
- Missing permission keys return `false` (not `undefined`).
- Stale context from a previous user is suppressed until the current session matches.

That's why the UI should branch on `ready` instead of guessing.

## Protect a page with permission state

When a whole page depends on one recordAccess, skip the manual branching:

```ts [app/pages/audit.vue]
import { workspaceAudit } from '~/convex/auth/permissions'

useAuthGuard({
  permission: workspaceAudit,
  redirectTo: '/',
})
```

`useAuthGuard` waits for the permission query to resolve before deciding. It doesn't race ahead of backend-owned policy.

## Route protection and permission guards do different jobs

Easy to collapse, costly to debug when you do:

| Layer                                  | Answers                                         | Used for                 |
| -------------------------------------- | ----------------------------------------------- | ------------------------ |
| `definePageMeta({ convexAuth: true })` | Is this user signed in?                         | Require a session at all |
| `useAuthGuard({ permission: ... })`    | Does the signed-in user have this recordAccess? | Gate a specific page     |

Authentication and authorization fail for different reasons. Model them separately.

## Common pitfalls

- Hard-coding `can` values in the context query instead of deriving them from backend guards. That defeats the projection.
- Treating projected permissions as a place to re-express record-level policy. Keep record checks in `authorize` or domain checks.
- Pointing `trellis.permissions.query` at the wrong export path. Trellis catches this; fix the path.
- Using a strict appIdentity that requires a tenant on pages where onboarding users still need to resolve (see [Caller and appIdentity](/docs/permissions/caller-and-app-identity) for the split).
- Treating `useAccess()` as a convenience cache — it's a projection of backend policy, not a separate store.

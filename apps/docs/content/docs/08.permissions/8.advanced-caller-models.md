::caution
This page is for teams that need more than one caller model. Most apps resolve everything to the user path and ignore services entirely — if that's you, you can skip this.
::

Trellis supports multiple caller models, but it still wants one protected business model. The rule that prevents drift: transport tells you _how_ the call arrived; appIdentity resolution tells you _what_ that caller means in the app.

## User and agent paths should converge

For most app code:

- Browser and server calls resolve to a `user` caller.
- Agent calls resolve to an `agent` caller.
- Both should usually resolve to the same app-owned appIdentity.

That keeps UI, server, and MCP traffic on one authorization system instead of three drifting ones. Keep `kind: 'agent'` in the transport identity where it belongs — don't turn the protocol into the business role model.

```ts
type Caller =
  | { kind: 'user'; subject: string }
  | { kind: 'agent'; agentId: string; provider: 'mcp' }
  | { kind: 'service'; serviceId: string }
```

## Service callers are different

Service callers aren't "another user flavor." They're a distinct trust boundary for internal services, schedulers, or verified webhook-style flows.

Define them through `defineServices(...)` with explicit access policy:

```ts [convex/services.ts]
import { defineServices } from '@lupinum/trellis/auth'

export const services = defineServices({
  billing: {
    access: {
      tables: ['subscriptions'],
      tenant: 'global',
    },
  },
})
```

Or with a derived tenant scope from args:

```ts
export const services = defineServices({
  webhook: {
    access: {
      tables: ['subscriptions'],
      tenant: 'derived',
      deriveTenant: ({ args }) => args.workspaceId,
    },
  },
})
```

| Field               | Meaning                                           |
| ------------------- | ------------------------------------------------- |
| `tables`            | Explicit list of tables this service may access   |
| `tenant: 'global'`  | Service runs outside the tenant model             |
| `tenant: 'derived'` | Service runs with a tenant derived from call args |

That's a different trust model than resolving an app appIdentity and running role checks. Use it when the caller genuinely shouldn't pretend to be an end user.

## Choose the right model

| Normal appIdentity path                                        | Service caller                                        |
| -------------------------------------------------------------- | ----------------------------------------------------- |
| Caller should behave like an app user                          | Caller is infrastructure or system-owned              |
| Action belongs under the same role and tenant model as the UI  | Access should be constrained by explicit table policy |
| One permission model across browser, server, and agent traffic | Call shouldn't pretend to be an end user              |

::note
If you're reaching for a service caller only to avoid modeling the right appIdentity for an agent or webhook caller, that's the wrong fix. Resolve the real delegated appIdentity and use the normal appIdentity path.
::

## What's shipped vs. what isn't

The settled core promise:

- Caller resolution.
- AppIdentity resolution.
- Agent callers flowing into the same protected app model.
- Runtime-enforced service access via `defineServices(...)`.

What the docs deliberately don't overclaim:

- Transport impersonation helpers as the main public model. Use explicit caller and `actingFor` forwarding where the runtime supports delegation, and keep services as a separate trust boundary.

## Common pitfalls

- Encoding business roles directly into transport callers. Role lives on the appIdentity.
- Treating service callers as convenient user impersonation. They're a separate trust model.
- Letting agent traffic drift onto a parallel authorization system. Converge it with the user path.
- Using a service caller to skip resolving an appIdentity. That's policy debt, not simplification.

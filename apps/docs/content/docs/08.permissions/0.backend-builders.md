Backend builders are where a Convex function enters the Trellis runtime. Pick
the lane first, then write the handler.

```ts [convex/functions.ts]
import { defineTrellis } from '@lupinum/trellis/backend'

import { mutation as generatedMutation, query as generatedQuery } from './_generated/server'

export const { query, mutation } = defineTrellis({
  query: generatedQuery,
  mutation: generatedMutation,
})
```

`query` and `mutation` are lane containers. They are not callable root builders.
Every handler declares its trust lane at the call site.

## Pick the lane first

| Lane                                         | Use it when                                                      |
| -------------------------------------------- | ---------------------------------------------------------------- |
| `query.public(...)`                          | Anyone may read this data                                        |
| `mutation.public(...)`                       | Anyone may perform this bounded public write                     |
| `query.protected(...)`                       | The read depends on an app appIdentity or guard                  |
| `mutation.protected(...)`                    | The write depends on an app appIdentity, guard, or authorization |
| `query.unsafe(...)` / `mutation.unsafe(...)` | The handler intentionally leaves the normal protected shape      |

The old shape "leave out the guard and maybe it is public" does not exist in
current Trellis. Public is explicit. Protected is explicit. Unsafe is explicit.

## Public handlers

Public handlers are for data and writes that truly do not need a signed-in
caller or app appIdentity.

```ts [convex/features/todos/domain.ts]
export const list = query.public({
  args: {},
  handler: async (ctx) => {
    return await ctx.db.query('todos').order('desc').collect()
  },
})
```

Use public lanes for the smallest public starter path. Do not use them to avoid
auth setup. If the answer depends on who is calling, use a protected lane.

## Protected handlers

Protected handlers run the full Trellis backend path:

1. caller
2. appIdentity
3. guard
4. load
5. authorize
6. handler

```ts [convex/features/todos/domain.ts]
export const create = mutation.protected({
  args: createTodo.args,
  guard: isAuthenticated,
  handler: async (ctx, args) => {
    const appIdentity = await ctx.appIdentity()

    return await ctx.db.insert('todos', {
      ownerId: appIdentity.userId,
      title: args.title,
      completed: false,
      createdAt: Date.now(),
    })
  },
})
```

The guard answers the broad question. `load` and `authorize` answer the
record-specific question.

```ts
export const rename = mutation.protected({
  args: renameProject.args,
  guard: projectWrite,
  load: async (ctx, args) => {
    const project = await ctx.db.get(args.projectId)
    requireRecord(project, 'Project')
    return { project }
  },
  authorize: {
    label: 'project belongs to appIdentity workspace',
    check: (appIdentity, { project }) => project.workspaceId === appIdentity.workspaceId,
  },
  handler: async (ctx, args) => {
    await ctx.db.patch(args.projectId, { name: args.name })
  },
})
```

Missing appIdentity wiring is a setup error. A resolved `null` appIdentity is normal
unauthorized state unless the guard explicitly accepts anonymous callers.

## Unsafe handlers

Unsafe lanes are review-sensitive escape hatches. They require typed permit
metadata.

```ts
import { unsafe } from '@lupinum/trellis/backend'

export const generateUploadUrl = mutation.unsafe({
  permit: unsafe.permit({
    kind: 'preTenantUpload',
    reason: 'Generate upload URLs before a tenant-scoped asset record exists.',
    scope: ['assets'],
  }),
  args: {},
  handler: async (ctx) => {
    requireAuth(await ctx.appIdentity())
    return await ctx.storage.generateUploadUrl()
  },
})
```

Prefer the narrower escape when possible:

```ts
const db = ctx.db.escapeIsolation({
  reason: 'Agency dashboard spans multiple client workspaces.',
})
```

If the protected handler shape still works and only the database tenant boundary
needs to be crossed, keep `mutation.protected(...)` and use
`ctx.db.escapeIsolation({ reason })` inside it.

## Operations are the next step, not the default

Most writes should stay as normal protected mutations. Reach for an operation
when the business action needs preview, confirmation, stable projection across
surfaces, or destructive MCP access.

```ts
export const remove = mutation.protected(removeTodoOp)
export const previewRemove = mutation.protected(previewOf(removeTodoOp))
```

See [Operations](/docs/permissions/operations) when a normal handler stops being
the clearest shape.

## Common pitfalls

- Writing `query({ ... })` or `mutation({ ... })` instead of an explicit lane.
- Treating a missing guard as public.
- Using `mutation.unsafe(...)` to avoid modeling an appIdentity or tenant.
- Hiding `ctx.db.escapeIsolation(...)` in a helper where reviewers cannot
  see why the boundary changed.
- Wrapping every write in an operation just because operations exist.

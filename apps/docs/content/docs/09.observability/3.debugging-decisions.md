When something's denied, drifted, or unexpectedly crossed a trust boundary — this is the page. The point of Trellis observability isn't more logs; it's faster answers to specific questions.

The four questions this page helps you answer:

- Why was this denied?
- Which layer denied it?
- Did this cross a tenant boundary on purpose?
- Did a destructive confirmation drift between preview and execute?

## Debug in this order

::steps{level="3"}

### Find the correlation ID

Every request has one. Grab it from the client error, server error, your observability store, or the DevTools auth panel's latest decision trace.

### Find the first identity event

`caller.resolved` or `appIdentity.missing`. That tells you whether the runtime even agreed on who called.

### Find the first deny or error event

Whichever came first sets the direction of the rest of the investigation.

### Check for trust-boundary events before it

If `unsafe.handler.used` or `db.escape_isolation.used` fired, the handler intentionally crossed one of the default guardrails — the issue is probably in the escape-hatch logic, not the default one.

::

That sequence usually localizes the problem to one layer.

## Fast path in local development

When Trellis DevTools is enabled locally, start in the auth panel:

- **Projected inventory** shows the current static permission booleans from the backend-owned context.
- **Latest decision trace** shows the newest correlated request path, including caller kind, appIdentity kind, tenant, denial explanation, and the event sequence.

That is the shortest route from "button is disabled" or "request was denied" to the actual runtime decision without digging through raw event payloads first.

## Common authorization failures

If you see one of these, the next question is "which layer decided?":

| Event              | The decision came from                                                        |
| ------------------ | ----------------------------------------------------------------------------- |
| `guard.denied`     | The outer boundary guard — the caller didn't even reach `load`                |
| `authorize.denied` | The resource-bound rule — `load` ran, then the per-resource check rejected    |
| `rls.denied`       | Runtime tenant/service row enforcement — the row wasn't valid for this caller |

Each one points to a different fix:

- `guard.denied` → the caller is wrong (wrong role, wrong tenant) or the guard is wrong.
- `authorize.denied` → the resource-bound policy rejected correctly; either the user lacks access or the rule needs adjusting.
- `rls.denied` → the handler crossed a tenant/service data boundary that the runtime rejected.

## Trust-boundary checks

If behavior looks broader than expected, look for:

- `unsafe.handler.used` — someone declared the handler with `unsafe.*`.
- `db.escape_isolation.used` — someone used `ctx.db.escapeIsolation({ reason })`.
- `service.access.checked` / `service.access.denied` — a service caller path evaluated access.

::tip
`db.escape_isolation.used` isn't a warning — it's a fact. The question is whether the handler _should_ have crossed. If yes, fine. If no, the fix is to remove the escape hatch and model the appIdentity or tenant correctly.
::

## Destructive drift

If a destructive tool or operation fails after preview, follow this trail:

| Event                         | Reading                                                 |
| ----------------------------- | ------------------------------------------------------- |
| `operation.preview.started`   | Preview began                                           |
| `operation.preview.completed` | Preview returned (capture the state)                    |
| `operation.confirm.validated` | Client came back with a matching confirmation           |
| `operation.confirm.drifted`   | Client's confirmation no longer matches previewed state |

`confirm.drifted` means the underlying data changed between preview and execute — Trellis refused to run a destructive action against drifted state. That's the system working as designed; the UX fix is to re-preview rather than override.

For MCP tools, also check the confirmation mode. `backend` mode emits the final confirm/execute events from Convex operation execution. `transport` mode emits them from the MCP transport after it redeems the token and before it calls the transport-confirmed projection.

## MCP failures

For tool problems, the sequence is:

| Event           | Meaning                                       |
| --------------- | --------------------------------------------- |
| `tool.called`   | Agent invoked the tool                        |
| `tool.denied`   | RecordAccess gating rejected before execution |
| `tool.executed` | Tool ran to completion                        |
| `tool.failed`   | Tool ran and threw                            |

If the issue is recordAccess gating, the denial explanation (`policy`, `reason`, `suggestedAction`) usually tells you more than the raw transport error.

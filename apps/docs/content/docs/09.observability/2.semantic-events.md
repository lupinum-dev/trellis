Reference for the Trellis event vocabulary. Every event carries a `transport`, a `status`, and a `family`, plus family-specific fields.

## Core dimensions

| Dimension   | Values                                                               |
| ----------- | -------------------------------------------------------------------- |
| `transport` | `browser`, `nuxt-server`, `convex`, `mcp`, `service`, `webhook`      |
| `status`    | `success`, `deny`, `error`, `skip`                                   |
| `family`    | `identity`, `authorization`, `trust`, `operations`, `mcp`, `browser` |

## Identity and authorization

| Event                  | When it fires                                                |
| ---------------------- | ------------------------------------------------------------ |
| `caller.resolved`      | Transport identity resolved                                  |
| `appIdentity.resolved` | App appIdentity resolved from caller                         |
| `appIdentity.missing`  | Caller arrived but no appIdentity could be resolved          |
| `guard.allowed`        | Outer boundary guard passed                                  |
| `guard.denied`         | Outer boundary guard rejected                                |
| `authorize.allowed`    | Resource-bound authorization passed                          |
| `authorize.denied`     | Resource-bound authorization rejected                        |
| `rls.denied`           | Runtime tenant or service row enforcement blocked the access |

These tell you who the runtime believes the caller is and where authorization failed.

## Trust-boundary events

| Event                      | When it fires                                     |
| -------------------------- | ------------------------------------------------- |
| `unsafe.handler.used`      | Handler was declared with `unsafe.*`              |
| `db.escape_isolation.used` | Handler used `ctx.db.escapeIsolation({ reason })` |
| `service.access.checked`   | Service access rule evaluated                     |
| `service.access.denied`    | Service access rule denied the call               |

High-signal events for review-sensitive seams. If a handler quietly started crossing tenants, this is how you find out.

## Operation events

| Event                         | When it fires                                  |
| ----------------------------- | ---------------------------------------------- |
| `operation.preview.started`   | Preview phase began                            |
| `operation.preview.completed` | Preview phase returned a result                |
| `operation.confirm.validated` | Confirmation payload matched the preview       |
| `operation.confirm.drifted`   | Confirmation no longer matches previewed state |
| `operation.execute.completed` | Handler ran to completion                      |
| `operation.execute.failed`    | Handler threw                                  |

Most useful for destructive flows and operation-backed MCP tools. A `confirm.drifted` is the signal that something changed between preview and confirm — and Trellis wouldn't let the execute run.

## MCP events

| Event                        | When it fires                                  |
| ---------------------------- | ---------------------------------------------- |
| `tool.called`                | Agent called a tool                            |
| `tool.denied`                | Tool visible but denied by recordAccess/policy |
| `tool.confirmation.required` | Destructive tool asked the agent to confirm    |
| `tool.executed`              | Tool completed successfully                    |
| `tool.failed`                | Tool execution threw                           |

These tell you whether the tool was visible, denied, awaiting confirmation, or actually executed.

## Browser and runtime events

| Event                                     | When it fires             |
| ----------------------------------------- | ------------------------- |
| `auth.session.checked`                    | Client checked auth state |
| `query.subscribed` / `query.unsubscribed` | Subscription lifecycle    |
| `query.updated`                           | Query received new data   |
| `query.failed`                            | Query errored             |
| `mutation.completed` / `mutation.failed`  | Mutation lifecycle        |
| `action.completed` / `action.failed`      | Action lifecycle          |
| `upload.completed` / `upload.failed`      | Upload lifecycle          |
| `connection.lost` / `connection.restored` | Websocket transitions     |

## Reason codes

Denial and error events carry a `reason` field. Common values:

| Reason                                                                 | Meaning                                              |
| ---------------------------------------------------------------------- | ---------------------------------------------------- |
| `guard.auth_required`                                                  | Signed-in caller needed, none present                |
| `guard.denied`                                                         | AppIdentity failed the outer guard                   |
| `authorize.denied`                                                     | AppIdentity failed the resource-bound rule           |
| `rls.denied`                                                           | Runtime tenant/service row enforcement denied access |
| `service.access.denied`                                                | Service access rule blocked it                       |
| `tool.recordAccess_denied`                                             | Agent lacked the recordAccess                        |
| `tool.disabled`                                                        | Tool exists but is disabled                          |
| `tool.confirmation_mismatch`                                           | Destructive confirmation didn't match preview        |
| `tool.execution_failed`                                                | Tool handler threw                                   |
| `query.failed` / `mutation.failed` / `action.failed` / `upload.failed` | Runtime surfaces for failures                        |
| `operation.execute.failed`                                             | Operation handler threw                              |

## Denial explanations

Denial events carry a structured explanation:

| Field             | Purpose                                         |
| ----------------- | ----------------------------------------------- |
| `decision`        | Which rule decided                              |
| `message`         | Human-readable line                             |
| `policy`          | Policy identifier (guard label, authorize name) |
| `workspaceId`     | Tenant context at decision time                 |
| `suggestedAction` | One concrete next step                          |

Suggested actions are deliberately small and concrete:

- `sign_in`
- `invite_to_workspace`
- `grant_recordAccess`
- `switch_tenant`
- `retry_with_confirmation`
- `contact_admin`

UI code can branch on these instead of parsing error strings.

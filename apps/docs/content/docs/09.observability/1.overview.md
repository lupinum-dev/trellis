Trellis observability isn't "more logs." It's a vocabulary of decisions — every denial, every trust-boundary crossing, every destructive preview/confirm cycle emits a structured event that explains _why_, not just _what_.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    observability: {
      service: 'my-app',
    },
  },
})
```

Semantic observation is enabled by default, but the default sink does not ship
events to a production log vendor for you. Configure this block when you want to
label the service, narrow capture, change the verbosity floor, or disable it with
`enabled: false`. Core owns the event vocabulary, redaction, correlation, and
test capture; delivery to your observability stack is intentionally outside the
default runtime.

## What Trellis actually traces

| Family          | Examples                                                              | Why it matters                  |
| --------------- | --------------------------------------------------------------------- | ------------------------------- |
| Identity        | `caller.resolved`, `appIdentity.resolved`, `appIdentity.missing`      | Who the runtime believes called |
| Authorization   | `guard.allowed`, `guard.denied`, `authorize.denied`, `rls.denied`     | Which layer allowed or denied   |
| Trust boundary  | `unsafe.handler.used`, `db.escape_isolation.used`, `service.access.*` | Every intentional boundary hop  |
| Operations      | `operation.preview.*`, `operation.confirm.*`, `operation.execute.*`   | Destructive flow integrity      |
| MCP             | `tool.called`, `tool.denied`, `tool.executed`, `tool.failed`          | Agent-facing traffic            |
| Browser/runtime | `query.*`, `mutation.*`, `upload.*`, `connection.*`                   | Client-side state transitions   |

The shape lets you ask decision-level questions — _who was denied, by which guard, on which resource_ — instead of stitching log lines by timestamp.

`rls.denied` is the event name for runtime tenant/service row enforcement denials. It does not imply a public custom-RLS authoring model.

## Correlation is built in

Events share correlation context across boundaries:

- Browser → Nuxt server → Convex → MCP → services, webhooks.

A denial in one layer stays connected to the request that started it. You don't have to correlate log fragments by timestamp and hope.

::note
The correlation header is configurable. Default is `x-trellis-correlation-id`; set your own if your infra already uses something like `traceparent`.
::

## Configure it at the module level

```ts [nuxt.config.ts]
trellis: {
  observability: {
    enabled: true,
    // capture: { backend: true, mcp: true, browser: false },
    // level: 'critical',
    // sample: { browser: 0.1 },
    // correlation: { header: 'traceparent' },
    // service: 'my-app',
    // explainability: { agentDenials: true },
  },
}
```

| Option               | Controls                                           |
| -------------------- | -------------------------------------------------- |
| `capture`            | Which surfaces emit (browser, server, Convex, MCP) |
| `level`              | Verbosity floor                                    |
| `sample`             | Per-event-family sample rates                      |
| `correlation.header` | Header name for request correlation                |
| `service`            | Service label on emitted events                    |
| `explainability`     | Explainability (reason codes, suggested actions)   |

Production defaults keep backend and MCP observation enabled, browser capture
off, and the level at `critical`. Development defaults are more verbose so local
decision debugging works without extra setup. In both modes, treat Trellis as the
semantic event source, not as your log pipeline.

See [Configuration: module options](/docs/configuration/module-options) for the full shape.

## Not a generic logging pitch

Trellis doesn't try to replace your app's logging. The value is the event _vocabulary_ — a denial is `guard.denied` with `reasonCode: 'guard.denied'`, not a free-form string your log aggregator has to regex for.

Trellis is the opinionated path for repeated apps on the same stack:

- Nuxt
- Convex
- Better Auth
- MCP

It is not a generic integration helper.

## Fastest on-ramp

Create a real app root first:

```bash
pnpm dlx @lupinum/trellis init my-app --template public
```

Or start with signed-in users:

```bash
pnpm dlx @lupinum/trellis init my-app --template personal
```

Or start directly on the workspace baseline:

```bash
pnpm dlx @lupinum/trellis init my-app --template workspace
```

If the workspace app also needs MCP from day one:

```bash
pnpm dlx @lupinum/trellis init my-app --template workspace-mcp
```

CMS product setup is owned by the integration package, not the Trellis starter
surface.

Then install dependencies, verify the generated app, and start the local loop:

```bash
cd my-app
pnpm install
pnpm dlx @lupinum/trellis doctor
pnpm dev:local
```

That is the first Trellis loop: choose the smallest starter that matches the
product, keep its fixture-backed shape, use explicit backend lanes, and run the
doctor before changing the generated wiring.

## Pick the smallest real baseline

| Starting point      | Pick it when                                            | Read first                                                                                         |
| ------------------- | ------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **Public-only app** | You want SSR and live queries with no sign-in flow yet. | [`01-public-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/01-public-todo)       |
| **Personal app**    | You need signed-in users but not tenant-scoped data.    | [`02-auth-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/02-auth-todo)           |
| **Workspace app**   | You need roles, tenant data, or trusted server callers. | [`03-team-workspace`](https://github.com/lupinum-dev/trellis/tree/main/examples/03-team-workspace) |
| **Workspace MCP**   | You need the workspace baseline plus agent tools.       | [`workspace-mcp` starter](./5.canonical-app-layout.md#workspace-mcp)                               |

::tip
If you are unsure, choose the smaller baseline. Trellis should let simple apps stay simple.
::

## Who Trellis is for

Use Trellis when:

- you are already building on Nuxt + Convex + Better Auth
- one backend model needs to serve browser UI, Nitro routes, and maybe MCP
- isolation, roles, permission projection, or destructive workflows are part of the real app
- you want the CLI, examples, and guardrails to push the team toward one consistent shape

## Who should skip it

Skip Trellis when:

- a tiny public or one-off internal app does not need this much structure
- you want a neutral helper layer instead of a framework
- you plan to discard the canonical layout immediately
- the app does not need shared auth, permission, or MCP conventions across surfaces

## Day-one mental model

You only need three ideas up front:

1. Shared input contracts live in `shared/features/<name>/contract.ts`.
2. Backend feature code lives under `convex/features/<name>/`.
3. UI-specific code can live under `app/features/<name>/`, with route shells staying thin.

Treat each feature folder as one app capability. The shared contract names the
inputs, Convex owns the policy and data, and Nuxt owns presentation.

Everything else is a deeper layer, not day-one homework.

## What success looks like

The first visible win is still small:

1. generate the smallest matching starter
2. run `pnpm install`
3. run `pnpm dlx @lupinum/trellis doctor`
4. start `pnpm dev:local`
5. render one query and see the page update without writing refetch glue

From there, Trellis grows into auth, permissions, operations, and MCP without changing the app’s core shape.

## Add layers only when needed

- **Auth** when the app needs signed-in callers
- **Permissions** when the browser needs backend-owned recordAccess projection
- **Operations** when destructive work needs preview/confirm
- **MCP** when the same backend must serve an agent client

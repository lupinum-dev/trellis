Use this page to check whether your app still matches the Trellis shell the CLI generates.

The tree below reflects what `pnpm dlx @lupinum/trellis init <name> --template public|personal|workspace|workspace-mcp` writes today.

## Rule

If you start with the CLI, keep the generated shape.

Trellis is meant to make apps boring to navigate. The point is not to preserve every starter file forever. The point is to keep the shell stable and put business code where it changes together.

## Canonical core

```text
nuxt.config.ts
convex/
  functions.ts
  schema.ts
  features/
shared/
  features/
app/
  features/
  pages/
server/
  api/
  mcp/
```

Every starter converges on that feature-first core. RecordAccess-specific folders are added only when the starter needs them:

- auth-enabled starters add `convex/auth.ts`, `convex/auth.config.ts`, `convex/convex.config.ts`, `convex/http.ts`, `convex/test.setup.ts`, and `convex/auth/`
- permission-enabled starters add `convex/permissions/`
- MCP-enabled workspace apps add `server/mcp/` files and `server/middleware/mcp-auth.ts`

This is the load-bearing contract:

- `nuxt.config.ts` owns module wiring.
- `convex/auth.ts`, `convex/auth.config.ts`, `convex/http.ts`, and `convex/convex.config.ts` own auth and Convex host wiring when auth is enabled.
- `convex/functions.ts` owns the app’s backend entry builders.
- `convex/schema.ts` owns tables and indexes.
- `convex/auth/` owns app identity decisions when auth is enabled.
- `convex/permissions/` owns frontend permission projection when permissions are configured.
- `convex/features/` owns backend feature modules.
- `shared/features/` owns input contracts shared by Convex, Nuxt, and MCP.
- `app/features/` owns feature-scoped UI code.
- `server/api/` owns app server routes.
- `server/mcp/` owns MCP projection.

Each top-level feature component should keep the route shell thin and gather the
UI that talks to one backend feature in one place.

## Starter baselines

All official starters share the feature-first core. They differ in which recordAccess folders they populate first.

### `public`

Use this when the app has public data only.

Generated first:

- `shared/features/todos/contract.ts`
- `convex/features/todos/{schema,domain,index}.ts`
- `app/features/public/components/PublicStarterPage.vue`

This starter does not generate auth or permission folders.

### `personal`

Use this when the app has signed-in users but no tenant model yet.

Generated first:

- `shared/features/todos/contract.ts`
- `convex/features/todos/{schema,domain,index}.ts`
- `convex/features/users/{schema,index}.ts`
- `app/features/personal/components/PersonalStarterPage.vue`

### `workspace`

Use this when the app needs roles, isolation, or backend-owned permission projection.

Generated first:

- `convex/auth/caller.ts`
- `convex/features/index.ts`
- `shared/features/todos/contract.ts`
- `convex/features/todos/{schema,permissions,domain,feature,index}.ts`
- `convex/features/workspaces/{schema,domain,feature,index}.ts`
- `convex/features/users/{schema,feature,index}.ts`
- `convex/permissions/context.ts`
- `app/features/workspace/components/WorkspaceStarterPage.vue`

### `workspace-mcp`

Use this when the app needs the workspace baseline plus MCP projection.

Additional generated files:

- `convex/features/mcpKeys/domain.ts`
- `server/lib/mcp-invalid-bearer-throttle.ts`
- `server/middleware/mcp-auth.ts`
- `server/mcp/index.ts`
- `server/mcp/runtime.ts`
- `server/mcp/tools/*`

## Where new code goes

Keep the structure boring:

- New backend modules go under `convex/features/<name>/`
- New shared input contracts go under `shared/features/<name>/`
- New feature UI goes under `app/features/<name>/`
- New guard and appIdentity logic goes under `convex/auth/`
- New permission projection goes under `convex/permissions/`
- New MCP tool files go under `server/mcp/tools/`

Do not scatter the same concern across ad-hoc `lib/`, `utils/`, and parallel backend folders.

## Stable now

The official Trellis starter surface today is:

- `pnpm dlx @lupinum/trellis init <name> --template public|personal|workspace|workspace-mcp`
- `pnpm dlx @lupinum/trellis add mcp|uploads|operation|entity`
- canonical root shell plus `convex/features|shared/features|app/features`
- optional MCP surface under `server/mcp/`

CMS product setup is owned by the integration package. Trellis keeps advanced component and bridge
examples, but does not expose CMS as a beginner app starter.

Only the listed starter set is part of the current product contract.

## Practical test

If two Trellis apps take more than a minute to orient, the platform shape is still too loose.

## Common pitfalls

- Keeping the root shell but scattering new business code into ad-hoc `lib/`, `services/`, or parallel backend folders.
- Treating the starter output as disposable scaffolding instead of the baseline shape the rest of the docs assume.
- Copying files from a richer starter when the app only needs the smaller baseline.

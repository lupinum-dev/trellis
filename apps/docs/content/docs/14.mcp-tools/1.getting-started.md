Expose one MCP tool end to end. This page walks the narrowest safe path: an endpoint, middleware that authenticates bearer tokens, a runtime, and a read-only tool. Destructive tools come later — they have their own requirements.

If you are starting a new app, start from the workspace MCP starter:

```bash
pnpm dlx @lupinum/trellis init my-app --template workspace-mcp
```

That gives you the workspace baseline plus the MCP projection files. If you are
adding MCP to an existing workspace app, use this page as the checklist and keep
the same rule: MCP projects the backend model; it does not become a second
backend.

::steps{level="2"}

### Define the MCP endpoint

```ts [server/mcp/index.ts]
export default defineMcpHandler({
  name: 'mcp-reference',
  browserRedirect: '/',
})
```

Small handler. The runtime does the work.

### Authenticate requests in middleware

```ts [server/middleware/mcp-auth.ts]
export default defineEventHandler(async (event) => {
  if (!event.path?.startsWith('/mcp')) return

  const header = getHeader(event, 'authorization')
  if (!header?.startsWith('Bearer ')) return

  const token = header.slice('Bearer '.length)
  const hash = await sha256(token)

  const mcpAuth = await validateMcpToken(event, hash)
  if (mcpAuth) event.context.mcpAuth = mcpAuth
})
```

The pattern: read `Authorization: Bearer`, hash, validate the hash in Convex, put the result on `event.context.mcpAuth`. Tokens stay hashed at rest; the runtime gets a clean transport identity.

### Define the MCP runtime

```ts [server/mcp/runtime.ts]
import { defineMcpApp } from '@lupinum/trellis/mcp'
import { createServerConvexCaller } from '#trellis/server'

import { getMcpCaller, getMcpDelegation } from './auth'

export const mcpRuntime = defineMcpApp({
  callConvex: async (event, { caller, actingFor }) =>
    createServerConvexCaller(
      event,
      caller.kind === 'agent'
        ? {
            auth: 'trusted',
            caller,
            actingFor,
          }
        : { auth: 'none' },
    ),
  resolveCaller: async (event) => getMcpCaller(event),
  resolveActingFor: async ({ event }) => getMcpDelegation(event),
  resolveAccess: async ({ caller, convex }) => {
    // derive tool-level recordAccess
  },
})
```

| Field              | Does                                                    |
| ------------------ | ------------------------------------------------------- |
| `callConvex`       | Per-request Convex caller bound to the MCP caller       |
| `resolveCaller`    | Turns middleware context into a caller                  |
| `resolveActingFor` | Optionally resolves the user subject the agent acts for |
| `resolveAccess`    | Coarse recordAccess snapshot for tool gating            |

The usual shape is:

- `caller` = the MCP agent, for example `agent:mcp_key_123`
- `actingFor` = the represented user, for example `user:u_42`

That keeps “who is calling” separate from “who they are acting for”.

If your app does not have represented-user semantics for MCP, omit `resolveActingFor` entirely. Trellis only forwards actingFor when you opt into it.

### Export the tool factory

```ts [server/mcp/runtime.ts]
export const tool = mcpRuntime.tool
```

Individual tools import this.

### Start with a read tool

```ts [server/mcp/tools/list-todos.ts]
import { api } from '#trellis/api'
import { listTodos } from '~~/shared/features/todos/contract'

import { tool } from '~/server/mcp/runtime'

export default tool.query({
  schema: listTodos,
  call: api.features.todos.domain.list,
  meta: {
    name: 'list-todos',
  },
})
```

One read-only tool. The real business rules stay in the protected handler.

::

## Schema boundaries

Trellis keeps schema boundaries explicit on purpose:

- Convex validators are the protected backend contract.
- Edge-only Zod should live next to the route, page, or feature component that owns that boundary.
- `shared/features/` is for input contracts you intentionally share across Convex, Nuxt, and MCP.
- MCP tools project Convex args into an MCP-safe schema internally when Trellis registers the tool.

That means the MCP tool surface shares backend intent, not the exact frontend form schema.

## Why read-only first?

::caution
Destructive tools have extra requirements — they must be backed by an [operation](/docs/permissions/operations) with preview, and `destructiveOperations` must be configured. Getting the transport and auth path working with a read tool first keeps those concerns separate.
::

The safe order is:

1. read-only `tool.query(...)`;
2. bounded writes through `tool.mutation(...)` only when the called ref carries
   matching bounded-write safety metadata;
3. sensitive, destructive, audited, bulk, or external-side-effect work through
   `tool.operation(...)`.

## Common pitfalls

- Starting with a destructive tool before the read-only transport and auth path works.
- Treating the MCP bearer token as business authorization instead of the transport identity input.
- Reusing frontend form validation as if it were the MCP tool contract.

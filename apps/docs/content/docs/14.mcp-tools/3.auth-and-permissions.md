MCP recordAccess checks keep the tool surface understandable. They don't replace business authorization — the protected Convex handler still owns the real decision. Two layers, each doing a different job.

| Layer                      | Owns                                   | Failure mode                         |
| -------------------------- | -------------------------------------- | ------------------------------------ |
| Transport (MCP middleware) | Who the bearer token represents        | No identity → no caller              |
| RecordAccess snapshot      | Coarse tool visibility and callability | Tool hidden or pre-rejected          |
| Backend authorization      | The actual business decision           | Same `guard` / `authorize` as the UI |

## Transport identity first

MCP requests resolve a transport-shaped [caller](/docs/concepts/glossary#caller) — usually an `agent` caller derived from the validated bearer token and middleware context.

That's transport identity. It tells the runtime _how_ the call arrived. It doesn't decide what the caller can do.

::warning
Public MCP means deliberately public and read-only. Scoped workspace tools, writes, destructive tools, session mutation, and code-mode endpoints require bearer authentication.
::

## RecordAccess snapshot second

`defineMcpApp(...)` computes a recordAccess snapshot for coarse tool gating:

- Whether the tool should appear in the agent's tool list.
- Whether the tool should be callable at all.

That keeps the MCP surface predictable. If the agent doesn't have a recordAccess, the tool either isn't visible or is pre-rejected. But that's just the outer layer.

::note
Trellis now re-checks MCP `permission` and `enabled(...)` at execution time as well. Discovery still shapes the tool list, but execution does not trust session-time visibility alone.
::

## Business authorization still lives in Convex

This is the rule that keeps the system honest:

::warning
**Tool recordAccess checks don't replace backend authorization.** The same protected Convex handler the UI calls is the handler the MCP tool calls. Guards, isolation, and authorize phases all still run. The recordAccess gate is a fence; it isn't the building.
::

Trellis examples derive coarse MCP recordAccess, then still call protected Convex handlers that enforce the real rules through actors, guards, isolation, and `authorize`.

## Why the split

| Without the split                                | With the split                               |
| ------------------------------------------------ | -------------------------------------------- |
| Tool layer accumulates business policy over time | Tool layer keeps policy surface minimal      |
| Agent path drifts from UI path                   | Agent path converges with UI path            |
| Two models to keep in sync                       | One backend model, projected to two surfaces |

The browser and the agent end up on the _same_ authorization system. That's the design goal.

## Common pitfalls

- Treating recordAccess visibility as the final authorization decision.
- Letting the MCP tool layer grow its own business-policy branches instead of calling the protected handler.
- Forgetting that execution-time permission checks still matter even if discovery hid the tool correctly.

Two related paths to define a tool, depending on whether you need the app runtime:

| API                                           | When                                                          |
| --------------------------------------------- | ------------------------------------------------------------- |
| `defineTool(...)`                             | Custom read, diagnostic, or external-service tool             |
| `tool.query(...)` from `defineMcpApp(...)`    | Read tool uses recordAccess snapshots and runtime auth        |
| `tool.mutation(...)` from `defineMcpApp(...)` | Bounded write tool backed by explicit safety metadata         |
| `tool.operation(...)`                         | Sensitive, destructive, audited, or external-side-effect work |

For most app-backed tools, use the runtime factory. Static permission handles and caller handling live there. Standalone `defineTool(...)` is not an app-write path.

MCP tools are allowlisted. Trellis registers tools you explicitly declare or
generate from explicit inventory; it does not expose Convex refs just because a
file exists in a directory.

MCP recordAccess checks are advisory. They decide whether a tool should be visible
or attempted, but the called backend handler still runs caller, appIdentity, guard,
tenant, load, authorize, and handler logic. If backend authorization denies after
MCP allowed the tool, Trellis returns that backend denial and records a redacted
recordAccess/backend drift observation.

Server/MCP files import shared descriptors and permission keys. They should not
import Convex-side permission implementations or handler implementation modules.
The backend ref still enforces the real policy.

## Schema boundary

`schema` in a Trellis MCP tool is the backend-facing Convex args contract. Trellis projects that schema into the Zod raw shape that `@nuxtjs/mcp-toolkit` expects when the tool is registered.

That projected MCP schema is transport metadata, not your form schema. Browser forms and Nitro `/api/*` handlers should keep their own hand-authored Zod validation next to the edge boundary that owns it. Promote shapes into `shared/features/` only when they are intentionally runtime-neutral.

## Common tool shape

A tool definition picks:

| Field                  | Purpose                                            |
| ---------------------- | -------------------------------------------------- |
| `meta.name`            | Tool identifier shown to the agent                 |
| `meta.description`     | Agent-facing description                           |
| `call`                 | The underlying Convex function                     |
| factory lane           | `tool.query`, `tool.mutation`, or `tool.operation` |
| `schema`               | Args validator                                     |
| `permission`           | Coarse static permission gate                      |
| `rateLimit` (optional) | Per-tool request budget                            |
| `mapResult` (optional) | Shape the return value for the agent               |

Example:

```ts [server/mcp/tools/list-todos.ts]
import { api } from '#trellis/api'
import { listTodos, todoReadKey } from '~~/shared/features/todos/contract'

import { tool } from '~/server/mcp/runtime'

export default tool.query({
  schema: listTodos,
  call: api.features.todos.domain.list,
  permission: todoReadKey,
  rateLimit: { max: 20, window: '1m' },
  meta: {
    name: 'list-todos',
  },
})
```

Direct `tool.query(...)` may point at a raw Convex read ref when the tool file
owns the public MCP schema, permission, and result mapping. Direct
`tool.mutation(...)` refs must carry Trellis metadata from public/protected
backend builders or shared descriptors so write safety is checked at startup.

`tool.mutation(...)` is only for bounded, non-destructive writes whose backend
metadata agrees with that safety class. The tool file can confirm or narrow the
classification, but it cannot make a sensitive, destructive, audited, bulk, or
external-side-effect backend handler look safe.

## Tool rate limits

`rateLimit` is per tool bucket and stays scoped to the current caller where the runtime has one.

Distributed enforcement is explicit. If you want MCP rate limits to hold across multiple instances, wire a shared atomic store into the runtime:

```ts [server/mcp/runtime.ts]
import Redis from 'ioredis'
import { createRedisMcpRateLimitStore, defineMcpApp } from '@lupinum/trellis/mcp'

const redis = new Redis(process.env.REDIS_URL!)

export const mcpRuntime = defineMcpApp({
  rateLimitStore: createRedisMcpRateLimitStore({
    client: redis,
  }),
  // ...
})
```

Without `rateLimitStore`, Trellis uses process-local memory. That is fine for local/dev and tests, not for distributed production enforcement.

Standalone `defineTool(...)` also accepts `rateLimitStore` when you are not using `defineMcpApp(...)`.

`trellis doctor` statically recognizes the first-party Redis helper. If you wire a custom store, doctor treats it as unverified and leaves the atomicity review to you.

::note
Tool `rateLimit` is an **outer guard** for MCP traffic. It is the right place to slow down agent spam and reject bursts before more work starts. It is not a replacement for application-layer quotas inside Convex.
::

For user, workspace, or billing-tier quotas inside your app, use an application-layer limiter in the called Convex function. Trellis recommends the Convex rate-limiter component for that inner policy layer. See [Rate-limiting layers](/docs/concepts/rate-limiting-layers).

## Start with the common-case tools

| Good first tools                   | Save for later                               |
| ---------------------------------- | -------------------------------------------- |
| Read-only query tools              | Destructive actions (delete, bulk update)    |
| Bounded writes via `tool.mutation` | Dynamic session-bound tool registration      |
| Clear single-purpose actions       | Tools that compose multiple business actions |

Destructive tools belong after the rest works. See [Destructive tools](/docs/mcp-tools/destructive-tools).

## Custom tools

Use standalone `defineTool(...)` only for custom read/diagnostic behavior or
external-service calls that do not write app state. It lives on the advanced
subpath so first-reader docs stay on the blessed `mcp.tool.*` lanes:

```ts
import { unsafe } from '@lupinum/trellis/mcp'
import { defineTool } from '@lupinum/trellis/mcp/advanced'

export default defineTool({
  schema,
  effect: 'external-service',
  permit: unsafe.permit({
    kind: 'externalService',
    reason: 'Calls a diagnostic external service without app writes.',
    scope: ['mcp'],
  }),
  handler: async (_args, ctx) => ctx.ok({ ok: true }),
})
```

`defineTool(...)` exposes `ctx.query(...)`, but not `ctx.mutation(...)` or
`ctx.action(...)`. App writes belong in `tool.mutation(...)` or
`tool.operation(...)`.

## Permission gates

The permission projection from `defineMcpApp(...)` lets you gate tool visibility or execution:

- **Visibility gate** — tool doesn't appear in the agent's tool list.
- **Execution gate** — tool appears but rejects at call time.

::note
Permission gating keeps the tool surface understandable. It isn't business authorization — the protected Convex handler still owns the real permission decision. See [Auth and permissions](/docs/mcp-tools/auth-and-permissions).
::

## Common pitfalls

- Putting business authorization into the MCP tool definition instead of the protected Convex handler.
- Importing Convex permission implementations into server/MCP files instead of shared permission keys.
- Declaring a direct mutation as bounded in the tool file when the backend metadata says it is sensitive, destructive, audited, bulk, or external-side-effect work.
- Reusing browser-form Zod as the tool schema instead of treating the Convex args contract as the backend seam.
- Shipping tool `rateLimit` in production without an explicit distributed `rateLimitStore`.

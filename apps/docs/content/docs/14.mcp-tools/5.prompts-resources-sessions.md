Tools are the main surface, but MCP has three more: prompts, resources, and sessions. Each one solves a different problem — and none of them replace backend authorization.

| Surface   | Use it for                                                    |
| --------- | ------------------------------------------------------------- |
| Prompts   | Structured guidance (plan before mutating)                    |
| Resources | Durable read-only context (guides, policy docs)               |
| Sessions  | Per-connection state (preferences, dynamic tool registration) |

## Prompts

Trellis re-exports MCP prompt helpers so prompt files can live alongside tools.

Good use: planning a runbook workflow before mutating anything, so the agent considers the shape of the work before acting on it.

::note
Prompts are structured _guidance_, not policy. They don't replace backend authorization. An agent following a prompt still hits the same guards and authorize phases as a direct call.
::

## Resources

Resources expose durable read-only context:

- App guides.
- Inventories.
- Policy descriptions.

The MCP reference example exposes a resource with a Markdown guide for the available MCP recordAccess. Good shape for context the agent should read before acting.

## Sessions

When `mcp.sessions` is enabled, `useMcpSession()` gives you a storage-backed namespace scoped to the MCP session ID:

| Good fit                              | Bad fit                                         |
| ------------------------------------- | ----------------------------------------------- |
| Saved focus or preference state       | Durable app data (use Convex)                   |
| Per-session dynamic tool registration | Cross-session user state (use your users table) |

Sessions are connection-scoped. Don't use them as a database.

If auth state, role, or recordAccess changes, don't treat the current session inventory as durable policy. Keep execution-time authorization authoritative, and let the next MCP session rebuild tools and resources against fresh state.

## Server registration

`useMcpServer()` returns the active MCP server instance during a live MCP request. From there you can register or remove tools, prompts, and resources dynamically.

This is the advanced path for per-session or per-request server surfaces. Most apps don't need it — stick to static tools.

## Common pitfalls

- Using prompts or resources as if they were authorization instead of agent context.
- Storing durable application state in MCP sessions instead of Convex.
- Rebuilding core tool policy inside dynamic session registration instead of keeping execution-time authorization in the backend.

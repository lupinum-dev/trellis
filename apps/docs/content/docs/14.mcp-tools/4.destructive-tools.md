Destructive MCP tools use operation-backed tooling, not raw one-shot mutations.
That way preview, confirmation, and execute all tie to the _same_ protected
business action — the agent can't accidentally run a destructive mutation
without first seeing what it's about to do.

The accepted 1.0 shape is explicit checked binding: import the shared operation
descriptor, then bind the generated preview and execute refs. A future one-liner
is only safe when the imported value is a shared descriptor or generated
operation handle. Do not import Convex implementation modules into server/MCP
tool files.

```ts [server/mcp/tools/delete-runbook.ts]
import { executeOperationRef, previewOperationRef } from '@lupinum/trellis/backend'

import { api } from '#trellis/api'
import { removeRunbookDescriptor } from '~/shared/features/runbooks/contract'
import { tool } from '~/server/mcp/runtime'

export default tool.operation(removeRunbookDescriptor, {
  execute: executeOperationRef(removeRunbookDescriptor, api.features.runbooks.domain.remove),
  preview: previewOperationRef(
    removeRunbookDescriptor,
    api.features.runbooks.operations.previewRemove,
  ),
  previewOperation: 'mutation',
  meta: {
    name: 'delete-runbook',
  },
})
```

One tool, one underlying operation, three refs wired through Trellis.

::note
`execute` and `preview` must be the real operation projections for that operation.
If the refs do not carry matching operation metadata, Trellis rejects the binding at startup.
::

## Confirmation modes

`tool.operation(...)` supports two confirmation boundaries:

| Mode        | Use when                                                 | Execute ref                                    |
| ----------- | -------------------------------------------------------- | ---------------------------------------------- |
| `backend`   | The Convex operation execution should verify the token   | `executeOperationRef(operation, ref)`          |
| `transport` | The MCP server verifies and redeems before a bridge call | `transportExecuteOperationRef(operation, ref)` |

`backend` is the default. It forwards `_confirmationToken` into the execute ref, so direct Convex destructive operation execution stays strict.

Use `transport` for component-backed or bridge-backed MCP tools where the MCP
route owns confirmation confirmation before calling a package or bridge boundary.
Production `transport` tools require a distributed `confirmationStore`; the
in-memory store is development-only. The execute ref must be a transport
execute projection that requires a trusted `operation-execute` forwarding
envelope, so the backend only accepts calls that crossed the MCP transport
boundary.

```ts [server/mcp/tools/publish-page.ts]
import { previewOperationRef, transportExecuteOperationRef } from '@lupinum/trellis/backend'

import { api } from '#trellis/api'
import { publishPageDescriptor } from '~/shared/features/pages/contract'

export default tool.operation(publishPageDescriptor, {
  execute: transportExecuteOperationRef(
    publishPageDescriptor,
    api.features.pages.domain.publishAction,
  ),
  preview: previewOperationRef(publishPageDescriptor, api.features.pages.domain.previewPublish),
  confirmationMode: 'transport',
})
```

## Why this is required

Trellis binds destructive tools to three things:

| Requirement                                        | Enforced because                                           |
| -------------------------------------------------- | ---------------------------------------------------------- |
| An operation with a stable `id`                    | Confirmation payload references the operation              |
| An `execute` ref projected from that operation     | Guarantees execute runs the same handler preview described |
| A `preview` ref projected from that same operation | Prevents a different preview from misleading the agent     |

::warning
If the execute and preview refs don't match the operation's projections, Trellis rejects the binding. That prevents a destructive tool from drifting away from the operation it claims to represent — the kind of bug that would otherwise surface as "the agent confirmed X and the system did Y."
::

The binding also checks operation metadata: operation id, kind, args/schema
metadata, permission key, preview ref, and execute ref must agree. Confirmation
then binds to the operation identity and previewed state. It does not replace
execute-time guard, load, authorize, tenant binding, or drift checks.

## The confirmation flow

::steps{level="3"}

### Preview

Agent calls the preview. Trellis runs the operation's preview phase against current state.

### Present summary and confirmation state

Preview returns the Trellis operation preview envelope: `allowed`, `summary`,
`blockers`, `warnings`, `effects`, `details`, and the machine-checked
`confirm` payload. The MCP response exposes the user-facing confirmation at
`preview.confirmation.token` only when `allowed` is true and there are no
blockers.

### Confirm

Agent sends the confirmation payload back to Trellis.

### Execute

Trellis validates the confirmation matches the previewed state, then runs the
execute path. In `backend` mode Convex re-runs the operation's policy and drift
checks from the confirmation token before the handler mutates data. In
`transport` mode MCP owns confirmation confirmation and drift validation, while
the Convex transport execute ref requires the trusted `operation-execute`
forwarding envelope and still runs the operation handler behind the bridge
boundary.

::

If the confirmation drifts (state changed between preview and confirm), Trellis denies the execute. The agent re-previews instead of overriding.

## Common pitfalls

- Binding a destructive tool directly to a mutation instead of an operation-backed preview and execute pair.
- Treating the preview summary as advisory text instead of the state the later confirmation must match.
- Reusing one operation's preview with a different operation's execute ref.
- Importing the Convex operation implementation into the MCP tool file instead of a shared descriptor or generated operation handle.
- Treating confirmation as permission to skip backend authorization.

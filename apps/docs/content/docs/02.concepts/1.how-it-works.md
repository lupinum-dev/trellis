Trellis rests on one claim: browser UI, server routes, webhooks, and agents should hit **one** app-owned business layer — not four parallel backends that drift out of sync. This page explains the model that makes that claim possible.

If you want the vocabulary only, go to the [Glossary](/docs/concepts/glossary). This page is the narrative; the glossary is the dictionary.

## The core sequence

Every call — from anywhere — follows the same five-step shape:

1. A [**caller**](/docs/concepts/glossary#caller) says _how_ the call arrived.
2. An [**appIdentity**](/docs/concepts/glossary#appIdentity) says _who_ that caller is in your app.
3. [Tenant](/docs/concepts/glossary#tenant) rules decide what data exists for that appIdentity.
4. [Guards](/docs/concepts/glossary#guard) and [authorization](/docs/concepts/glossary#authorize) decide whether the work is allowed.
5. [Operations](/docs/concepts/glossary#operation) bundle reusable business actions where preview-and-confirm matters.

Nuxt and MCP then project that same backend model into different transports. Everything else in Trellis is support around that shape.

## One business layer, many transports

The failure mode Trellis is trying to prevent: you ship a browser backend, then a webhook backend, then an internal server backend, then an agent backend — and six months later no two of them enforce the same rules. Permission drift, duplicated business logic, transport-specific exceptions you can't trust.

Trellis treats those four as **projections** of one app-owned business layer instead.

## Caller first, appIdentity second

This split is the part people try to skip. Don't.

Trellis separates two questions the app has to answer on every call:

- _How did this call arrive?_ → the [**caller**](/docs/concepts/glossary#caller) (transport-shaped: anonymous visitor, signed-in user, forwarded agent, trusted server caller).
- _Who is that caller in this app?_ → the [**appIdentity**](/docs/concepts/glossary#appIdentity) (app-shaped: role, workspace membership, tenant scope).

The first is resolved from transport signals. The second is resolved from _your_ data. Collapsing them — treating "it's a webhook so it can do anything" or "it's signed in so it must be an admin" — is how transport identity leaks into business policy, and that's where bugs live.

## The three runtimes

Trellis has three pillars, each a projection of the same model into a different surface.

The interesting part isn't that Trellis can speak MCP. It's that agent access is treated as part of the app model — same guards, same actors, same authorization path — rather than as a separate side system you bolt on and hope stays in sync.

## What changes by transport — and what doesn't

Transports still differ in real ways: how identity is resolved, whether the route is public or internal, whether confirmation is required, what result envelope you return. That's fine, and Trellis doesn't flatten those differences.

What doesn't change is the **business authorization path**. If a browser caller, a webhook caller, and an agent caller can all reach the same business action, they all pass through the same Trellis-owned enforcement.

::warning
Transport visibility is not authorization. A hidden MCP tool, a private server route, a webhook signature — these are useful, but they don't replace the guards on the handler itself. Transport controls gate _reach_; guards gate _permission_. You need both.
::

Signed identity forwarding follows the same rule. A valid `_trellisForwarding`
envelope proves the call crossed an approved server/MCP/bridge boundary. It does
not grant permission, skip appIdentity resolution, or replace `guard`, `load`,
`authorize`, and `handler`.

## Why operations matter

[Operations](/docs/concepts/glossary#operation) are the reusable seam for destructive and preview-requiring work: admin UI actions, confirmation dialogs, server-side workflows, webhook-triggered business actions, destructive agent tools. Define the flow once; every caller runs the same checks and the same preview.

Not every mutation should be an operation. But when a flow needs [preview](/docs/concepts/glossary#preview), confirmation, reuse across projections, or stronger safety around destructive work, operations are the right primitive.

## What Trellis does not own

Your product logic. Trellis gives you the structure; you fill in the rules.

- The appIdentity model (what a valid user looks like in your app)
- Guards and checks (what counts as permission)
- Table shape and tenant boundaries
- Business behavior (what each handler actually does)
- What an operation means semantically

Trellis owns the machinery _around_ those decisions so they stay coherent across every transport.

## When this model matters most

You can ignore most of this page when the app is public-only. The model becomes load-bearing the moment you add _any_ of:

- auth
- tenant boundaries
- protected mutations
- server callers
- agent tooling

That's when "one business layer" stops being architecture-speak and starts preventing real, observable drift.

## Common pitfalls

- Treating the transport as the business role model (_"it's signed in, so it's an admin"_).
- Forking server or agent code paths into their own special-case business handlers.
- Relying on tool visibility or webhook signatures as if they were authorization.
- Re-reading this page instead of going back to the task guide once the model has clicked.

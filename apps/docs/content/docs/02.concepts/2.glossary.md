Every term Trellis uses in more than one guide is defined here. Pages link to these anchors instead of redefining terms inline, so you always land on the canonical meaning.

The terms cluster into four groups: **identity**, **authorization**, **structure**, and **transports**.

::note
This page is reference, not onboarding. A basic app does not need every term here on day 1. Start with [Start here](/docs/getting-started/start-here) and come back only when a guide sends you here.

## Identity

### Caller

Transport identity — _how_ a call arrived. Typical shapes: an anonymous browser visitor, a signed-in user session, a forwarded agent request, a trusted server caller. A caller is resolved from transport signals (session cookies, tokens, signatures) and nothing more. It deliberately knows nothing about roles, tenants, or business policy.

### AppIdentity

App-owned business identity — _who_ the caller is in your app. Carries things like `role`, `workspaceId`, workspace membership. Two different principals (a browser session and an MCP agent request) can resolve to the same appIdentity, and that's the point: business policy stays the same regardless of transport.

### Permission appIdentity

A relaxed form of appIdentity used only for frontend access context. Returns a shape even for anonymous users so the UI can compute projected `can(...)` results without branching. Distinct from the stricter appIdentity used in backend handlers.

## Authorization

### Guard

A coarse, reusable authorization check defined with `defineGuard(label, check)`. Guards answer outer-boundary questions — _is there a signed-in caller?_, _is this appIdentity a workspace member?_, _does this appIdentity hold the admin role?_ They compose with `.and(...)`, `.or(...)`, `.not()` and get attached to handlers as `guard: ...`.

### Check

An appIdentity-aware boolean predicate — the atomic unit of policy. Guards are one kind of check (attached to a handler via `guard: ...`); resource-bound predicates like `canUpdateTodo(todo)` are another. Evaluate a check against the current appIdentity with `can(...)`.

### Load

The handler phase that fetches the data a downstream decision depends on. Runs after `guard`, before `authorize`. Common pattern: load the record, confirm it exists with `requireRecord`, return it in the context object.

### Authorize

The resource-bound authorization phase that runs against loaded data. Use it when the decision to allow an action depends on the thing being acted on — ownership, state, relationships. Keeps resource-specific policy out of the coarse entry guard.

### Operation

A reusable business action created with `defineOperation(...)`. An operation bundles `guard`, `load`, `authorize`, `preview`, and `handler` into one unit so the same action can be projected cleanly into a mutation, an MCP tool, and a confirmation UI without logic drift.

### Preview

The read-only projection of what an operation _would_ do, used to power confirmation UX for destructive work. Returns a display shape (`summary`, `warn`, `affects`) and a confirmation token that the real execution must check back against.

### Destructive

An operation `kind` that requires extra safety: a preview phase must exist, and the confirmation token must match before execution. Reserved for actions that lose or mutate data in ways you can't cheaply undo.

## Structure

### Business layer

The one app-owned business layer that every transport hits. One schema, one set of guards, one authorization path. Trellis exists to keep this layer from being forked per transport.

### Handler

The execution function of a query, mutation, action, or operation. Runs last in the phase chain (`guard → load → authorize → handler`) and does the actual work.

### Tenant

The isolation scope for multi-tenant data — typically a workspace, organization, or team. In Trellis, tenant boundaries are enforced at runtime, not just by convention.

### Isolation

Runtime enforcement, configured in `defineTrellis(...)`, that rewrites `ctx.db` queries to filter by the current appIdentity's tenant. Tables listed under `isolation.tables` get automatic scoping; the default field is `workspaceId`.

### Unsafe surface

The explicit escape-hatch family for review-sensitive paths. Trellis exposes `ctx.db.escapeIsolation({ reason })` for cross-scope data access and `query.unsafe(...)` / `mutation.unsafe(...)` for intentional handler-level bypasses. Using either one is an explicit decision, not the default.

## Transports

### Transport

How a call reaches the business layer — browser UI, Nitro server route, webhook, MCP agent. Transports differ in how identity resolves, how the result is shaped, and what protections they need. They do not differ in business authorization.

### Projection

A transport-facing surface (a Nuxt composable, a server helper, an MCP tool) that exposes the same business action in a way that transport can consume. Projections are thin: the logic stays in the business layer.

### App Runtime

The protected Convex business layer. Caller resolution, appIdentity resolution, tenant scoping, protected queries and mutations, actions, and operations. This is where Trellis' authorization model lives.

### Nuxt Runtime

The Nuxt-facing projection. SSR, hydration, composables (`useConvexQuery`, `useConvexMutation`), route protection, server helpers, upload helpers, access context.

### Agent Runtime

The agent-facing projection over MCP. Tool exposure, permission-aware tool discovery, destructive preview and confirmation, caller forwarding into the App Runtime.

Rate limiting is not one thing in a Trellis app. Different layers protect different problems.

If you only remember one rule, remember this:

- **Trellis owns MCP ingress rate limits**
- **Better Auth owns auth endpoint rate limits**
- **the consumer app owns business quotas inside Convex**

## The four layers

### 1. Network edge

Examples:

- CDN or WAF rules
- reverse proxy limits
- provider abuse protection

Use this for:

- floods
- obvious abuse
- infrastructure protection

This is **not** Trellis.

### 2. Auth endpoints

Examples:

- sign in
- password reset
- verification flows

Use this for:

- slowing down credential abuse
- protecting auth-sensitive routes

This is **Better Auth**. Trellis only passes the config through.

### 3. MCP ingress

Examples:

- `list-posts`
- `create-post`
- `bulk-delete-runbooks`

Use this for:

- agent or tool spam
- accidental bursts
- rejecting junk before more work starts

This is **Trellis**.

```ts [server/mcp/tools/list-todos.ts]
export default tool.query({
  schema: listTodos,
  call: api.features.todos.domain.list,
  permission: workspaceRead,
  rateLimit: { max: 20, window: '1m' },
  meta: {
    name: 'list-todos',
  },
})
```

That `rateLimit` is an outer guard on the MCP tool surface. It is not a business quota for every Convex endpoint behind the tool.

### 4. Business quotas inside Convex

Examples:

- per-user message caps
- per-workspace LLM token budgets
- invite or export throttles
- plan-tier usage limits

Use this for:

- fairness
- cost control inside your domain logic
- user or workspace-specific quotas

This is the **consumer app**.

## One request, two rate limits

For expensive MCP flows, using two layers is often the right design.

### Outer layer: Trellis MCP guard

Use Trellis `rateLimit` to slow down the tool itself.

### Inner layer: app-owned Convex quota

Inside the called Convex function, use an application-layer limiter to enforce business policy.

```ts [convex/features/todos/domain.ts]
export const create = mutation.protected({
  args: createTodo.args,
  guard: hasWorkspace.and(hasMinimumRole('member')),
  handler: async (ctx, args) => {
    const appIdentity = await ctx.appIdentity()

    await rateLimiter.limit(ctx, 'createTodo', {
      key: appIdentity.workspaceId,
      throws: true,
    })

    return await ctx.db.insert('todos', {
      workspaceId: appIdentity.workspaceId,
      title: args.title,
      completed: false,
      createdAt: Date.now(),
    })
  },
})
```

Trellis recommends the Convex rate-limiter component for this inner application-layer policy.

## Why not use just one limiter?

Because one limiter does not solve every layer well.

If you use only the inner Convex limiter:

- junk MCP traffic still reaches Convex
- you pay more Convex calls and I/O just to reject obvious bursts

If you use only the outer MCP limiter:

- you do not get per-user, per-workspace, or billing-tier quota logic inside the app

The right answer depends on the layer.

## Pricing story

This boundary also affects cost.

### Trellis MCP limiter

If Trellis rejects the request before Convex work starts:

- fewer Convex function calls
- less Convex I/O
- less downstream work

### App-layer Convex limiter

If you enforce quotas inside Convex:

- you get richer policy
- but you also spend Convex resources to evaluate that policy

That tradeoff is fine when you need business rules. It is wasteful when all you needed was a cheap outer guard.

## When should I use what?

Use **Better Auth** when the endpoint is part of auth.

Use **Trellis MCP `rateLimit`** when the problem is the MCP tool surface itself.

Use an **app-layer Convex limiter** when the rule belongs to your domain:

- user quota
- workspace budget
- plan limit
- expensive business action

## What Trellis does not do

Trellis does not provide a generic framework-wide `rateLimit` option for every `query(...)`, `mutation(...)`, `action(...)`, or `defineOperation(...)`.

That is intentional. Trellis owns the MCP transport layer, not every application quota policy.

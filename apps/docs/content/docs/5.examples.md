Pick the smallest example that proves the boundary you actually need.

## First reader path

Read these in order:

1. `01-public-todo`
2. `02-auth-todo`
3. `03-team-workspace`
4. `07-mcp-reference` only when you need the `workspace-mcp` branch

That's the cleanest ladder from minimal Trellis to the canonical protected-app reference.

Each one adds one major layer:

- `01` adds the smallest `defineTrellis(...)` runtime and one query and mutation path.
- `02` adds Better Auth and appIdentity resolution.
- `03` adds workspaces, permissions, and the canonical protected-app shell plus feature layout.
- `07` adds MCP projection after the workspace baseline is already clear.

If you only need a public or personal app, stop earlier. There's no benefit in inheriting workspace or agent complexity you don't need.

## Match the example to the problem

| Example                 | Open it when you need                                  | What it proves                                                 |
| ----------------------- | ------------------------------------------------------ | -------------------------------------------------------------- |
| `01-public-todo`        | Public-only queries and mutations                      | Smallest useful Trellis baseline                               |
| `02-auth-todo`          | Sign-in and personal ownership                         | Auth plus app-owned actors without tenancy                     |
| `03-team-workspace`     | Roles, permissions, or tenant boundaries               | Canonical protected workspace app                              |
| `04-saas-platform`      | Pagination, uploads, admin flows, practical SaaS depth | Canonical advanced workspace example                           |
| `05-visibility-access`  | Advanced visibility, redaction, enrollment, or sharing | Canonical advanced access-control example                      |
| `06-multi-workspace`    | Workspace switching or cross-scope operator views      | Canonical multi-workspace membership example                   |
| `07-mcp-reference`      | Workspace MCP starter path                             | Read tools, bounded writes, operation-backed destructive tools |
| `08-component-mini-cms` | Advanced packaged-integration boundaries               | Component bridge plus MCP over bridge refs                     |

## Pattern catalog after the first ladder

- `04-saas-platform`: pagination, uploads, Nitro routes, admin workflows.
- `05-visibility-access`: advanced visibility and sharing rules.
- `06-multi-workspace`: multi-workspace membership and cross-scope operator views.
- `07-mcp-reference`: workspace-MCP branch, after the workspace baseline is already clear.
- `08-component-mini-cms`: advanced package-integration reference, not beginner app setup.

Treat those local feature components as mini-Trellis boundaries: the route shell stays thin while the feature component owns one backend-facing seam.

The canonical examples use the same structural grammar when those surfaces exist:

- `convex/features/*` for backend business handlers
- `shared/features/*` for runtime-neutral contracts
- `app/features/*` for feature-scoped UI
- `convex/permissions/context.ts` for the configured permission query

## Canonical public example set

`examples/` is the canonical public example set today. Use it as the source of truth for:

- First-reader learning paths.
- Docs code references.
- Real public patterns worth copying.

`labs/` is a pressure suite for future direction. It is not part of the default public source of truth until something is promoted intentionally.

::note
`examples/` is the canonical public example set. `labs/` is archived or exploratory material, not a maintained public reference set.
::

## Read before you copy

Don't copy an example wholesale unless the app shape is genuinely close to yours. Instead:

1. Choose the smallest matching example.
2. Read the files that define the core boundary.
3. Copy only the pattern you need.
4. Stop before unrelated surfaces come with it.

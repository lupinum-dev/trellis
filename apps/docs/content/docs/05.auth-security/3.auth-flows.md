The concrete UI patterns for sign-in, sign-up, and sign-out. If the auth runtime is already wired ([Authentication](/docs/auth-security/authentication)), the rest is forms and redirects.

One rule keeps these pages clean: **Better Auth owns the auth action. Trellis owns the Convex refresh and the redirect that follows.** `useBetterAuthActions` is the seam between them.

## Sign up

```ts [app/components/SignUpForm.vue]
const client = useBetterAuthClient()
const authAction = useBetterAuthActions()

async function handleSignUp(payload: { name: string; email: string; password: string }) {
  if (!client) throw new Error('Auth client unavailable.')

  await authAction.execute(() => client.signUp.email(payload), { redirectTo: '/' })
}
```

## Sign in

Same shape, different Better Auth call:

```ts [app/components/SignInForm.vue]
const client = useBetterAuthClient()
const authAction = useBetterAuthActions()

async function handleSignIn(payload: { email: string; password: string }) {
  if (!client) throw new Error('Auth client unavailable.')

  await authAction.execute(() => client.signIn.email(payload), { redirectTo: '/' })
}
```

If the user landed on the sign-in page because [route protection](/docs/auth-security/route-protection) redirected them, the original path comes through as `?redirect=...` and takes priority over the `redirectTo` you pass here. That's how the two flows compose without fighting.

## Sign out

```ts [app/components/UserMenu.vue]
const { signOut } = useConvexAuth()

async function handleSignOut() {
  await signOut()
}
```

`signOut()` clears Convex auth first, then calls Better Auth sign-out. That order matters — the app stops acting authenticated _immediately_, before the network call returns.

## Tracked state on auth forms

`useBetterAuthActions` returns the same tracked-call shape as mutations: `status`, `pending`, `error`, `data`, `reset()`. Wire it straight into the form instead of inventing parallel refs:

```vue [app/components/SignInForm.vue]
<template>
  <form @submit.prevent="handleSignIn(form)">
    <UInput v-model="form.email" type="email" required />
    <UInput v-model="form.password" type="password" required />
    <UButton type="submit" :loading="authAction.pending.value">Sign in</UButton>

    <UAlert
      v-if="authAction.error.value"
      color="error"
      title="Authentication error"
      :description="authAction.error.value.message"
    />
  </form>
</template>
```

## Auth boundary components for the shell

The common layout for a public-facing auth page:

```vue [app/pages/auth/signin.vue]
<template>
  <ConvexAuthLoading>
    <p>Checking authentication…</p>
  </ConvexAuthLoading>

  <ConvexUnauthenticated>
    <SignInForm />
  </ConvexUnauthenticated>

  <ConvexAuthenticated>
    <p>You're already signed in. <NuxtLink to="/">Go home</NuxtLink></p>
  </ConvexAuthenticated>
</template>
```

Don't forget `definePageMeta({ skipAuthTokenFetch: true })` on this file — see [Route protection](/docs/auth-security/route-protection).

## Redirect precedence

When a sign-in completes, Trellis picks the redirect target in this order:

1. `?redirect=...` query param (from route protection).
2. `redirectTo` passed to `authAction.execute(...)`.
3. No redirect.

That's why you can set `redirectTo: '/'` as the fallback without stomping on the "send the user back where they came from" behavior.

## Common pitfalls

- Calling Better Auth directly (`client.signIn.email(...)`) and skipping `useBetterAuthActions` — the Better Auth session updates but Convex doesn't refresh.
- Re-implementing pending/error state outside the action — you end up with two sources of truth.
- Redirecting manually before the auth refresh finishes, so the next page still sees the old auth state.
- Treating sign-out as a Better Auth-only operation. It's a two-sided state transition.

Auth failures are almost always configuration bugs. Before reading page code, run the checks on this page in order — the first one catches most of them.

## Step 1: run doctor

```bash [terminal]
pnpm dlx @lupinum/trellis doctor
```

Doctor inspects the same assumptions the runtime makes:

- Nuxt app root, `@lupinum/trellis` installed, `convex` installed, module registered.
- `CONVEX_URL` (or `NUXT_PUBLIC_CONVEX_URL`) resolves to something.
- Auth env vars exist when `auth.enabled` is true.
- Better Auth routes are registered in `convex/http.ts`.

If doctor warns, fix that first. Don't open DevTools before the wiring is right.

## Step 2: env vars

When auth is enabled, these are the usual suspects:

| Var                  | What breaks when it's wrong                                                   |
| -------------------- | ----------------------------------------------------------------------------- |
| `CONVEX_URL`         | App boots against the wrong deployment                                        |
| `SITE_URL`           | Better Auth callbacks + trusted-origin checks fail                            |
| `CONVEX_SITE_URL`    | Token exchange fails when auto-derivation misses (custom HTTP Actions domain) |
| `BETTER_AUTH_SECRET` | Doesn't match the Convex deployment — signatures reject                       |

::tip
If auth works in one environment and not another, env mismatch is the default explanation until you've ruled it out. Diff your `.env` against the failing environment before anything else.
::

## Step 3: Better Auth route registration

The browser can render the form and post credentials while `convex/http.ts` is missing the route — what breaks is the token exchange that follows. Make sure this file exists and registers routes:

```ts [convex/http.ts]
import { httpRouter } from 'convex/server'

import { authComponent, createAuth } from './auth'

const http = httpRouter()
authComponent.registerRoutes(http, createAuth)

export default http
```

Symptom: sign-in "succeeds" but `useConvexAuth()` never flips to `isAuthenticated: true`.

## Step 4: redirect loops

If the browser bounces between the sign-in page and a protected page forever:

- The sign-in page doesn't have `definePageMeta({ skipAuthTokenFetch: true })`. Add it.
- The protected page's `redirectTo` points back at a protected page. Point it at a public auth route.
- A manual redirect fires before `authAction.execute(...)` finishes. Let the action own the redirect.

## Step 5: page code actually goes through the action

This is the subtle one. Calling the Better Auth client directly _looks_ like it works:

```ts
// ❌ Session updates on Better Auth side, but Convex auth doesn't refresh.
await client.signIn.email(payload)
navigateTo('/')
```

The user appears signed-in to the next page, then every protected query sees `null` and fails. Use the action:

```ts
// ✅ Convex refresh + redirect in one tracked flow.
await authAction.execute(() => client.signIn.email(payload), { redirectTo: '/' })
```

::warning
This is the #1 mysterious "auth works but queries say I'm not signed in" cause. Always go through `useBetterAuthActions`.
::

## Step 6: access context drift

If auth is fine but `useAccess()` returns `null` forever (or belongs to the wrong user), Trellis warns in development after a 2-second grace. Common causes:

- `trellis.permissions.query` points at the wrong Convex export.
- AppIdentity bootstrap returns `null` for the signed-in user (missing `users` row, bad `authKey` index).
- The permission query itself returns `null`.

Once auth is ready, Trellis trusts the server-projected access context. If `useAccess()` shows the wrong user or workspace, fix the access-context query or appIdentity resolver; the browser does not recompute that identity from Better Auth profile fields.

## Step 7: webhook / MCP traffic failing separately

If browser auth works but identity-forwarding or MCP surfaces fail, that's a
different trust path. Check `CONVEX_IDENTITY_FORWARDING_KEY` — it must exist in
local env **and** in the Convex deployment serving that traffic, and in
production it must be a long non-placeholder secret. Browser sign-in doesn't
touch it, so none of the earlier steps will catch the problem. The forwarding
path signs `_trellisForwarding`; it should not send raw forwarding fields or
identity-shaped public args.

## When to stop reading and re-run the checklist

Go back to step 1 when:

- A new environment was introduced.
- Auth started failing after changing callback or origin settings.
- Only server-side or MCP traffic is broken.
- One user works and another always gets `null` access context.

## Common pitfalls

- Skipping doctor and reading page code first.
- Copying env from another environment without updating `SITE_URL` / `BETTER_AUTH_SECRET`.
- Calling Better Auth directly and patching state after.
- Assuming identity-forwarding traffic shares the same auth path as browser sign-in.

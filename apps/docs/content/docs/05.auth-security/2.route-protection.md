Keep signed-out users out of a page before it renders. One line of route meta — no `v-if` gymnastics, no "render, wait, redirect" flash.

```ts [app/pages/dashboard.vue]
definePageMeta({
  convexAuth: true,
})
```

Trellis middleware sees that flag, checks the auth state, and redirects if the user isn't signed in. The page never paints for an unauthenticated visitor.

## Why this belongs in middleware

If the page renders and _then_ redirects from a `watch` or `onMounted`, the protection already leaked. The user saw the dashboard shell for a frame. Middleware runs before the page mounts, which is the only place protection can hold honestly while auth is still pending.

::warning
Protecting only with template `v-if="isAuthenticated"` isn't route protection — it's a conditional render. The page is still matched, still mounts, still fetches its data. Use route meta.
::

## Custom redirect target

If the sign-in page isn't at the default path, point to it:

```ts [app/pages/dashboard.vue]
definePageMeta({
  convexAuth: {
    redirectTo: '/auth/signin',
  },
})
```

When `preserveReturnTo` is on (see below), Trellis appends `?redirect=<current-path>` so the sign-in flow can send the user back.

## Set the default once, in module config

Most apps want the same redirect target across all protected pages. Configure it once:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: {
      enabled: true,
      routeProtection: {
        redirectTo: '/auth/signin',
        preserveReturnTo: true,
      },
    },
  },
})
```

Then pages just say `convexAuth: true`. Override per-page only when one route genuinely needs a different target.

## Opt out on auth pages

Sign-in and sign-up pages need to stay outside the auth path — otherwise they trigger token-fetch behavior for themselves, or redirect-loop when the middleware sees they're unauthenticated.

```ts [app/pages/auth/signin.vue]
definePageMeta({
  skipAuthTokenFetch: true,
})
```

Rule of thumb: any page _inside_ the auth flow uses `skipAuthTokenFetch`. Any page _behind_ auth uses `convexAuth: true`.

## Route protection ≠ authorization

This is the part that catches teams.

::caution
Route protection only answers "is the user signed in enough to _enter_ this page?" It does **not** answer "is the user allowed to read this row?" Backend [guards](/docs/concepts/glossary#guard) and [authorization](/docs/concepts/glossary#authorize) decide what a protected handler does. A user who sneaks past (bookmarked URL, delayed redirect, whatever) still can't read data the server's guards reject.
::

If the only thing stopping a user from seeing data is route protection, you have a permission bug. The server has to reject the request regardless of the URL the user typed.

## Common pitfalls

- Template-only protection (`v-if="isAuthenticated"`) with no route meta.
- Forgetting `skipAuthTokenFetch: true` on the sign-in page — causes redirect loops.
- Per-page `redirectTo` everywhere instead of one module-level default.
- Treating route protection as permission enforcement. Server guards are the real decision.

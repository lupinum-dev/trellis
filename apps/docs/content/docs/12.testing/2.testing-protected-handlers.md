When the thing under test is backend authorization, tenancy, or handler behavior — use the real handler path, seeded with real tenants and real users. That way the test proves the guard and authorize phases did what you expected, not just that some helper function returned the right value.

```ts [convex/posts.test.ts]
const ctx = createTestContext({
  schema,
  modules,
  tenant: { table: 'organizations', field: 'organizationId' },
})

const team = await ctx.seedTenant({
  name: 'Acme',
  users: {
    alice: { role: 'member' },
    bob: { role: 'viewer' },
  },
})

const postId = await team.users.alice.mutation(api.posts.create, {
  title: 'Seeded by helper',
  content: 'Goes through the real scoped mutation.',
})

await expect(team.users.bob.mutation(api.posts.publish, { id: postId })).rejects.toThrow(
  /Forbidden/,
)
```

Alice can create. Bob can't publish. The test goes through the real handler — guards and all.

## Seed tenants and users

```ts
const ctx = createTestContext({
  schema,
  modules,
  tenant: { table: 'organizations', field: 'organizationId' },
})

const team = await ctx.seedTenant({
  name: 'Acme',
  users: {
    alice: { role: 'member' },
    bob: { role: 'viewer' },
  },
})
```

That gives you authenticated callers with realistic tenant and role state. `team.users.alice` is an authenticated caller bound to this tenant — every `mutation(...)` / `query(...)` / `action(...)` goes through the full runtime.

Because the returned users are keyed from the object you passed to `seedTenant(...)`, the test stays readable without inventing a parallel factory layer just for callers.

## Test real protected mutations

```ts
const postId = await team.users.alice.mutation(api.posts.create, { title: '...', content: '...' })

await expect(team.users.bob.mutation(api.posts.publish, { id: postId })).rejects.toThrow(
  /Forbidden/,
)
```

This is the right level for permission rules. The test exercises the real handler path — if someone later adds an `authorize.check(...)` that's stricter than expected, this test catches it.

## Forward principals explicitly

When you need to prove parity between browser and forwarded callers (e.g. webhook, MCP):

```ts
const ctx = createTestContext({
  schema,
  modules,
  identityForwardingKey: 'test-only-shared-secret',
})

const identityForwarding = ctx.asCaller({
  kind: 'user',
  authKey: team.users.viewer.authKey,
  subject: `auth:${team.users.viewer.authKey}`,
})

const result = await identityForwarding.mutation(api.posts.comment, { postId, text: 'hi' })
```

Call the same protected handler; assert the permission result stays the same across transports. If browser and forwarded callers produce different outcomes, you have a transport-shaped authorization bug.

`asCaller(...)` is deliberately explicit. It does not try to infer app actors for you. The point of the helper is to test the transport boundary, not to hide it. It also requires the same identity-forwarding secret the real forwarded path uses, so the test cannot bypass verification by accident.

That `identityForwardingKey` value is a test harness signing secret. It signs the
same `_trellisForwarding` envelope shape as the real path; it is not a raw
forwarding field passed to your Convex handler.

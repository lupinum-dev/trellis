Test Trellis apps against the real runtime. The helpers give you a Vitest setup that works with Convex's edge runtime and a test context that seeds tenants, users, and principals — no hand-built mocks that bypass guards and authorization.

```ts [vitest.config.ts]
import { convexTestConfig } from '@lupinum/trellis/testing'

export default convexTestConfig({
  test: {
    include: ['convex/**/*.test.ts'],
  },
})
```

That's the whole Vitest side. Tests import `createTestContext(...)` for the per-test setup.

## The three main helpers

| Helper                         | Purpose                                           |
| ------------------------------ | ------------------------------------------------- |
| `convexTestConfig(...)`        | Vitest config wrapper with the edge-runtime setup |
| `createConvexTestModules(...)` | Module bridge for the generated Convex server     |
| `createTestContext(...)`       | Per-test seed + caller API                        |

## Create a test context

The high-value helper is `createTestContext(...)`:

```ts [convex/todos.test.ts]
import { createTestContext } from '@lupinum/trellis/testing'

const ctx = createTestContext({
  schema,
  modules,
})
```

From there:

- Seed data.
- Seed tenants and their users with `seedTenant(...)`.
- Call handlers through authenticated user callers returned by that seed.
- Forward explicit principals with `asCaller(...)` when you need to prove identity-forwarding parity between transports.

## What this is for

Use these helpers when you want realistic tests against _actual protected handlers_ instead of mocks that bypass the Trellis runtime model. A test that builds a fake appIdentity and calls an inner helper function is cheaper — and proves nothing about whether the handler's guard, load, or authorize phases do what you think.

::tip
If a test needs to reach around the Trellis runtime to work, that's a signal the test is at the wrong layer. Either move it up (integration test against the handler) or down (unit test on the helper itself).
::

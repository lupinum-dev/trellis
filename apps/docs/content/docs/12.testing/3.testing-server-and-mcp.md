When the thing under test crosses the Nuxt server or MCP boundary, pick the right layer for the assertion. Trellis ships focused contract tests for server helpers and integration harnesses for full transport flows — don't flatten them into one giant integration bucket.

## Server helper contract tests

Trellis has focused tests around:

- `serverConvexQuery`
- `serverConvexMutation`
- `serverConvexAction`

What they verify:

| Area                     | What's asserted                                               |
| ------------------------ | ------------------------------------------------------------- |
| Request shape            | Args, transport, headers routed correctly                     |
| Auth-mode handling       | `auto` / `required` / `none` / `trusted` behave as documented |
| Error reporting          | Helper name, function path, auth mode in errors               |
| Current-event resolution | Reads H3 event state correctly                                |

Right level for server-helper _contracts_. Not the right level for "does my full webhook flow work."

## MCP and integration flows

For MCP or end-to-end flows, use the internal harness and example suites. Good examples in this repo:

- MCP smoke coverage.
- Internal-harness server API tests.
- Example 07 MCP reference tests.

These exercise the whole pipeline (transport → runtime → handler → response). Building a one-off mock for a full pipeline usually ends up asserting that the mock works — not that the system does.

## Reset and seed helpers

The internal harness includes test-only Convex helpers for resetting and seeding state. Useful for repeatable MCP and server integration tests where isolated setup matters more than unit-level mocks.

## Rule of thumb

| Layer                                      | Use it for                         |
| ------------------------------------------ | ---------------------------------- |
| Plain unit tests                           | Pure helper logic, transformations |
| Trellis test context (`createTestContext`) | Protected backend behavior         |
| Server helper tests                        | Nitro helper contracts             |
| MCP / e2e suites                           | Full transport flows               |

::caution
Don't flatten these layers into one giant "integration test" bucket. Tests at the wrong layer either run slow and prove too much, or run fast and prove nothing.
::

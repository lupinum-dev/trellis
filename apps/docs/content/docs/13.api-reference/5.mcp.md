Reference for the public Trellis MCP runtime surface. Each tool definition projects the same backend authorization path as a browser call: transport changes, business rules do not.

MCP tools are allowlisted through explicit declarations or generated inventory.
Trellis does not expose Convex refs by directory convention.

## `defineMcpApp(options)`

Creates the MCP runtime and the tool factory bound to your Trellis backend model.

### Options

| Option                                     | Purpose                                                          |
| ------------------------------------------ | ---------------------------------------------------------------- |
| `callConvex(event, { caller, actingFor })` | Convex caller the MCP runtime uses for every tool invocation     |
| `resolveCaller(event)`                     | Map the transport request to a Trellis caller                    |
| `resolveActingFor?(...)`                   | Optional represented-user mapping for trusted forwarded calls    |
| `resolveAccess?(...)`                      | Optional tool-visibility / recordAccess snapshot layer           |
| `rateLimitStore?`                          | Optional distributed MCP rate-limit store for tool `rateLimit`   |
| `runtime?(...)`                            | Optional per-request runtime data passed to tool handlers        |
| `callerKey?(caller)`                       | Stable identity key for destructive confirmation + observability |
| `observability?`                           | MCP runtime observability options (transport, correlation IDs)   |

### Returns

| Export                               | Purpose                                                        |
| ------------------------------------ | -------------------------------------------------------------- |
| `tool.query(options)`                | Direct read tool factory                                       |
| `tool.mutation(options)`             | Direct bounded-write tool factory with backend safety metadata |
| `tool.operation(operation, options)` | Operation-backed tool factory for sensitive/destructive work   |

Direct query tools may point at raw Convex read refs when the tool owns its MCP
schema and policy. Direct mutation tools must point at Trellis public/protected
backend refs or shared descriptors that carry backend metadata.
`tool.mutation(...)` cannot down-classify a sensitive, destructive, audited,
bulk, or external-side-effect backend handler as a bounded write.

## Advanced custom tools

Standalone `defineTool(...)` and toolkit-level `defineMcpTool(...)` live under
`@lupinum/trellis/mcp/advanced`. They are for custom read, diagnostic, or
external-service handlers that cannot be represented as one Trellis-backed ref.
They are not an app-write lane. App-backed reads, bounded writes, and
destructive work should use `tool.query(...)`, `tool.mutation(...)`, and
`tool.operation(...)` from `defineMcpApp(...)`.

::note
`enabled` affects _tool visibility_ or _availability_, not backend business authorization. On the runtime-factory path, `permission` and resolved recordAccess shape discovery, but guards on the called handler still run on every call. If MCP visibility allows a tool and the backend later denies it, Trellis returns the backend denial and emits a redacted recordAccess/backend drift observation.
::

::note
`resolveActingFor` stays explicit. If you omit it, Trellis forwards no represented user.
::

## `tool.operation(operation, options)`

Preferred path for operation-backed tools, especially destructive ones. Accepts the same options as `defineTool` (except `schema`, `call`, `preview`, `operation`, `previewOperation` — those come from the operation itself).

### Options

| Option                                                                                                  | Type                                | Purpose                                                     |
| ------------------------------------------------------------------------------------------------------- | ----------------------------------- | ----------------------------------------------------------- |
| `execute`                                                                                               | Operation execute projection ref    | The mutation or action that runs the operation              |
| `preview?`                                                                                              | Operation preview projection ref    | Preview function reference (for destructive operations)     |
| `executeOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `execute`                                     |
| `previewOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `preview`                                     |
| `confirmationMode?`                                                                                     | `'backend' \| 'transport'`          | Where destructive confirmation is redeemed                  |
| `confirmationStore?`                                                                                    | `McpConfirmationStore`              | Required for production `transport` confirmation            |
| `confirmationTtlMs?`                                                                                    | `number`                            | Confirmation token lifetime; defaults to 5 minutes          |
| `schema?`                                                                                               | `ConvexSchema`                      | Optional schema override — defaults to the operation's args |
| _(plus everything in `defineTool` except `schema`, `call`, `preview`, `operation`, `previewOperation`)_ |                                     |                                                             |

### Semantics

- Destructive operations **require** a preview projection — binding fails without one.
- Destructive previews must return `OperationPreviewEnvelope`.
- Blocked previews (`allowed: false` or non-empty `blockers`) mint no confirmation token.
- Trellis binds confirmation tokens to operation identity **and** previewed state.
- Mismatched preview or execute refs are rejected at bind time, not at call time.
- `execute` and `preview` must be the real exported projections of the same operation.
- `backend` mode forwards `_confirmationToken`; Convex validates and redeems it.
- `transport` mode redeems confirmation in MCP, strips `_confirmationToken`,
  and requires the execute ref to receive a trusted `operation-execute`
  forwarding envelope.
- Confirmation does not replace execute-time guard, load, authorize, or tenant
  binding.

## Session and server helpers

| Helper            | Purpose                                                        |
| ----------------- | -------------------------------------------------------------- |
| `useMcpServer()`  | Access the active MCP server instance inside a request         |
| `useMcpSession()` | Per-session storage namespace (when `mcp.sessions` is enabled) |

## Result helpers

Normalize tool results into the expected structured MCP envelope:

| Helper                                  | Purpose                                  |
| --------------------------------------- | ---------------------------------------- |
| `wrapSuccess(data, summary?)`           | Success envelope                         |
| `wrapError(category, message, issues?)` | Error envelope                           |
| `wrapPreview(preview)`                  | Preview envelope (awaiting confirmation) |
| `withSummary(result, summary)`          | Add a summary line to a result           |

## Toolkit re-exports

Trellis re-exports MCP toolkit surfaces for composing the surrounding app:

- `defineMcpHandler`
- `defineMcpPrompt`
- `defineMcpResource`

Trellis-specific business alignment still lives in `defineMcpApp(...)` and the Trellis tool factory. When the tool layer needs a stable automation surface, project root handlers or bridge refs into the MCP runtime instead of duplicating business behavior in tool files.

`@nuxtjs/mcp-toolkit` owns MCP handler/session/runtime behavior. Trellis layers Convex caller construction, business-safe projection, structured result envelopes, and destructive confirmation semantics on top of those toolkit primitives.

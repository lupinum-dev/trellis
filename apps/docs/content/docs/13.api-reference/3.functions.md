Reference for the backend-facing Trellis surfaces used in Convex function files. Import these from `@lupinum/trellis/backend`.

For the authoring walkthrough, read [Backend builders](/docs/permissions/backend-builders). This page stays terse.

Trellis has one public authorization model: keep business authorization in `guard`, `load`, `authorize`, and `handler`, and let `isolation` plus `services` enforce infrastructure guardrails on `ctx.db`. Custom app-authored DB policy hooks are not part of the public authoring surface.

## `defineTrellis(...)`

Primary entrypoint for building protected query, mutation, and action builders on top of Convex.

```ts
defineTrellis(builders, options?)
```

### Options

| Option                  | Purpose                                                    |
| ----------------------- | ---------------------------------------------------------- |
| `caller`                | How transport identity is resolved                         |
| `actingFor`             | How represented-user forwarding is validated               |
| `appIdentity`           | How a resolved caller maps to the app appIdentity          |
| `isolation`             | Tables that stay tenant-scoped by default                  |
| `services`              | Explicit high-trust service callers                        |
| `observability`         | Semantic event emission config                             |
| `destructiveOperations` | Required before registering destructive operations         |
| `triggers`              | Convex triggers to install on the Trellis runtime path     |
| `onSuccess`             | Post-success side-work hooks for queries/mutations/actions |

### Returns

| Field                                                | Purpose                                                             |
| ---------------------------------------------------- | ------------------------------------------------------------------- |
| `query.public` / `query.protected`                   | Public and protected query lanes                                    |
| `mutation.public` / `mutation.protected`             | Public and protected mutation lanes                                 |
| `action.public` / `action.protected`                 | Public and protected action lanes when actions are wired            |
| `transportMutation(...)`                             | Trusted transport execute lane for bridge/MCP boundaries            |
| `internalQuery` / `internalMutation`                 | Internal lanes when internal builders are provided                  |
| `internalTransportMutation(...)`                     | Internal trusted transport lane when internal builders are provided |
| `query.unsafe` / `mutation.unsafe` / `action.unsafe` | Explicit handler-level escape-hatch lanes                           |

### DB surface

| Surface                              | Use                                                                                 |
| ------------------------------------ | ----------------------------------------------------------------------------------- |
| `ctx.db`                             | Default safe path                                                                   |
| `ctx.db.escapeIsolation({ reason })` | Explicit tenant-boundary bypass — keeps the protected handler shape                 |
| `query.unsafe` / `mutation.unsafe`   | Explicit handler-level bypass lanes — require `permit: unsafe.permit(...)` metadata |

See [Tenant escape and unsafe access](/docs/permissions/cross-scope-and-raw-access).

::note
Tenant and service enforcement happen in the runtime so handlers do not have to remember manual tenant filters on every read. That runtime layer is infrastructure, not a public app-authorization API.
::

## `defineOperation(...)`

Reusable business-action primitive for flows that project cleanly across UI, server, and MCP.

```ts
defineOperation({
  id,
  name,
  kind,
  args,
  guard,
  load,
  authorize,
  preview,
  handler,
})
```

### When to use

- Shared `guard`, `load`, and `authorize` across preview + execute.
- One action needs stable identity across transports.
- Destructive action that needs Trellis-enforced preview/confirm.

### Destructive requirements

::warning
These must be present for `kind: 'destructive'`:

- `kind` set to `'destructive'`.
- `id` on the operation.
- `preview(...)` function.
- `destructiveOperations` configured in `defineTrellis(...)` before registration.
  ::

Preview is side-effect-free: it reads, loads, authorizes, and computes display
plus confirmation material. Execute re-runs guard, load, authorize, tenant
binding, and drift checks before applying the handler. Preview success is not an
authorization grant.

### Projection helpers

| Helper                                         | Purpose                                                                                                                                  |
| ---------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `mutation.protected(operation)`                | Register the execute path                                                                                                                |
| `previewOf(operation)`                         | Produce the preview handler definition; register destructive previews with `mutation.protected(...)` when confirmation tokens are issued |
| `executeOperationRef(operation, ref)`          | Bind a backend execute ref to the operation metadata                                                                                     |
| `previewOperationRef(operation, ref)`          | Bind a backend preview ref to the operation metadata                                                                                     |
| `transportExecuteOperationRef(operation, ref)` | Bind a trusted transport execute ref for `confirmationMode: 'transport'` MCP tools                                                       |

## Handler context accessors

Protected handler context adds these Trellis surfaces:

| Accessor                             | Returns                                       |
| ------------------------------------ | --------------------------------------------- |
| `ctx.caller()`                       | Resolved transport identity                   |
| `ctx.appIdentity()`                  | App-owned appIdentity derived from the caller |
| `ctx.observe(...)`                   | Emits a semantic observability event          |
| `ctx.db`                             | Default database surface (tenant-scoped)      |
| `ctx.db.escapeIsolation({ reason })` | Explicit cross-scope surface                  |
| `query.unsafe` / `mutation.unsafe`   | Explicit handler-level escape hatch           |

::note
Caller and appIdentity are async runtime facts, not compile-time assumptions. `await ctx.caller()` every time you need it inside a handler.
::

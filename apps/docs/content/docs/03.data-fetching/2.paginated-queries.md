Render a cursor-paginated list without tracking cursors in your component. One composable owns the list, the next-page button, stale state during filter changes, and the point where you've run out of data.

```vue [app/pages/projects.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const {
  results: projects,
  status,
  loadMore,
  hasNextPage,
} = await useConvexPaginatedQuery(api.projects.list, {}, { initialNumItems: 12 })
</script>

<template>
  <ul>
    <li v-for="project in projects" :key="project._id">{{ project.name }}</li>
  </ul>
  <UButton v-if="hasNextPage" :loading="status === 'loading-more'" @click="loadMore(12)">
    Load more
  </UButton>
</template>
```

That's the default shape. Most of this page is about the states you render against and the moments you'd deviate.

## The states that matter

`status` drives the UI; cursor math doesn't show up in page code at all.

| Status               | Meaning                                  | UI usually shows              |
| -------------------- | ---------------------------------------- | ----------------------------- |
| `loading-first-page` | First page in flight                     | Skeleton or spinner           |
| `ready`              | One or more pages loaded; more may exist | The list + "Load more"        |
| `loading-more`       | Another page is loading                  | Disabled/spinning "Load more" |
| `exhausted`          | No more pages                            | Hide "Load more"              |
| `error`              | Pagination failed                        | Error state + retry           |
| `skipped`            | Args missing; query isn't running        | Nothing, or a gate            |

::tip
Don't flatten `ready` and `exhausted` into the same branch. Users read "Load more" disappearing as a signal that they've reached the end — that's worth the extra condition.
::

## Skip until the list can run

Paginated queries skip the same way plain queries do — through args, not by wrapping the composable in `v-if`.

```ts [app/pages/projects.vue]
const projectArgs = computed(() => (workspaceId.value ? {} : undefined))

const { results, status } = await useConvexPaginatedQuery(api.projects.list, projectArgs, {
  initialNumItems: 12,
})
```

When args are `undefined`: `status` is `'skipped'`, `results` is `[]`, and no subscription starts.

## Keep the list visible during filter changes

Default behavior clears `results` when args change. For workspace switches, filter swaps, or tab changes, that blank flash reads worse than showing the previous list while the next one loads:

```ts [app/pages/projects.vue]
const { results, isStale, status } = await useConvexPaginatedQuery(
  api.projects.list,
  () => ({ filter: filter.value }),
  { initialNumItems: 12, keepPreviousData: true },
)
```

During the transition `isStale` is `true`, `status` moves through `loading-first-page`, and the previous results stay rendered.

## Refresh vs. reset

Two different things, easy to confuse:

- **`refresh()`** — re-fetch the currently loaded pages. Same cursor position, same page count. Use after a write that should appear in the current view.
- **`reset()`** — start over from page one. Use when the filter or identity of the list changed and the previous position doesn't mean anything anymore.

## Turn off live subscription for snapshot lists

Most paginated lists should stay live. The exception is a report-style surface where manual refresh is more honest than background updates:

```ts [app/pages/exports.vue]
const { results, refresh } = await useConvexPaginatedQuery(
  api.exports.list,
  {},
  { initialNumItems: 25, subscribe: false },
)
```

::warning
`subscribe: false` is a commitment. New rows won't appear, deletions won't vanish, edits won't update — until someone calls `refresh()`. If the UI pretends otherwise, you'll chase "why is this stale?" bugs.
::

## Common pitfalls

- Managing cursors manually in the page — the composable already owns them.
- Forgetting `initialNumItems`, which leaves the first page size ambiguous.
- Treating `ready` and `exhausted` as the same branch, so "Load more" never actually goes away.
- Clearing results on every args change when `keepPreviousData: true` would read better.

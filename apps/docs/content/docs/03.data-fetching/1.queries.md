Fetch data from Convex with one line. It renders on the server, hydrates on the client, and stays live as the data changes — you don't have to refetch, poll, or wire subscriptions by hand.

```vue [app/pages/tasks.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const { data: tasks, pending, error } = await useConvexQuery(api.tasks.list, { status: 'active' })
</script>
```

That's the whole default path. The rest of this page is about when you'd want to deviate — skip a query, change args reactively, turn off the subscription, or keep old data visible during a transition.

Normal query execution failures do not throw from the composable path. They set `status` to `'error'` and populate `error.value`, so pages can render an inline error state while the same query can still recover on refresh or the next live update. Setup and configuration mistakes can still throw.

::note
Looking for the full option matrix? See [Composables](/docs/api-reference/composables). This page is the narrative.
::

## How it stays live

Three things happen in order, and you don't have to wire any of them:

1. SSR awaits the first result, so the HTML ships with real data.
2. That same value hydrates on the client — no refetch.
3. A Convex subscription takes over. When the underlying data changes, `data` updates and your template re-renders.

That's why `useConvexQuery` isn't just a fetch helper. It's the read path.

## Skip a query until it's ready

Pass `undefined` (or `null`) as the args when the query shouldn't run yet — for example, while auth is still loading.

```ts [app/pages/dashboard.vue]
const { isAuthenticated } = useConvexAuth()

const args = computed(() => (isAuthenticated.value ? {} : undefined))
const { data, status, pending } = await useConvexQuery(api.todos.list, args)
```

While skipped, the composable returns `status: 'skipped'`, `pending: false`, and no subscription starts. Prefer this over `v-if`-ing the composable itself — calling a composable conditionally breaks Vue's lifecycle rules.

## Reactive args re-subscribe automatically

Args can be a plain object, a `ref`, a reactive object, or a getter. When they change, Trellis cancels the old subscription and starts a new one.

```ts [app/pages/search.vue]
const tag = ref('alpha')

const { data, isStale } = await useConvexQuery(
  api.search.notes,
  () => ({ filter: { tag: tag.value } }),
  { keepPreviousData: true },
)
```

::tip
Use a getter (`() => ({ ... })`) when any field inside the args depends on another reactive value. Plain ref destructuring loses reactivity in practice.
::

## Keep old data visible during transitions

Default behavior clears `data` when args change. For search, filter swaps, or paginated refinements, that blank flash is worse than showing slightly stale data for a beat. Flip `keepPreviousData: true`:

```ts [app/pages/search.vue]
const { data, pending, isStale } = await useConvexQuery(
  api.search.notes,
  () => ({ query: term.value }),
  { keepPreviousData: true },
)
```

During the transition:

- the previous `data` stays visible;
- `pending` is `true`;
- `isStale` flips to `true` so the UI can show a subtle loading hint.

## One-shot reads (no subscription)

Most pages want live data. When they don't — an expensive snapshot, a request/refresh surface, a page that genuinely doesn't need to react to changes — turn the subscription off:

```ts [app/pages/reports.vue]
const { data, refresh } = await useConvexQuery(
  api.reports.snapshot,
  {},
  {
    subscribe: false,
  },
)
```

::warning
Don't use `subscribe: false` for data the UI expects to stay current. If someone else edits the underlying record, your page will silently go stale. This is the #1 cause of "why didn't the UI update?"
::

If you don't need composable state at all (no reactivity, no lifecycle), call `executeConvexQuery()` directly instead.

## Skip SSR for non-critical data

Use `server: false` when the query shouldn't block the initial render — typically dashboards, activity feeds, anything that's fine to appear a moment after the rest of the page.

```ts [components/ActivityFeed.vue]
const { data, pending } = await useConvexQuery(
  api.activity.list,
  {},
  {
    server: false,
  },
)
```

Before the first client result, `status` is `pending` and `pending` is `true`. The template should cope with an empty state — put a skeleton in.

## Default value + transform

Need a placeholder shape, or a derived view of the result? `default` seeds `data`; `transform` maps whatever comes back.

```ts [app/pages/notes.vue]
const { data } = await useConvexQuery(
  api.notes.list,
  {},
  {
    default: () => [{ _id: 'placeholder', title: 'Loading...' }],
    transform: (items) => items.map((n) => ({ ...n, title: n.title.toUpperCase() })),
  },
)
```

The transform applies to `default` as well, so keep the two shapes aligned.

## Common pitfalls

- Calling the composable conditionally (`if (x) useConvexQuery(...)`) instead of passing skip args.
- Using `subscribe: false` on data the UI needs to stay live, then chasing ghost "stale data" bugs.
- Clearing the UI on every arg change instead of `keepPreviousData: true`.
- Calling `useConvexQuery` in middleware or a non-component scope — use server helpers or `executeConvexQuery()` there.

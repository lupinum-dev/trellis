Show an offline banner, a reconnecting indicator, or a "changes are still syncing" hint — without rolling your own heuristics on top of raw websocket events. One composable exposes the signals the UI actually needs.

```ts [app/layouts/default.vue]
const { shouldShowOfflineUi, isReconnecting, pendingMutations } = useConvexConnectionState()
```

```vue [app/layouts/default.vue]
<template>
  <UAlert v-if="shouldShowOfflineUi" color="warning">
    You're offline. Changes will sync when you reconnect.
  </UAlert>
  <slot />
</template>
```

That's the intended use. The rest of this page is about which signal to reach for and which ones to leave alone.

## What's in the composable

| Key                   | What it tells you                                       |
| --------------------- | ------------------------------------------------------- |
| `state`               | Raw phase: `connecting`, `connected`, `reconnecting`    |
| `isConnected`         | Websocket is connected right now                        |
| `isReconnecting`      | Was connected, isn't now — actively trying to come back |
| `pendingMutations`    | Count of mutations queued on the client                 |
| `shouldShowOfflineUi` | The safe signal for offline banners (see below)         |

## Why `shouldShowOfflineUi` is not just `!isConnected`

The naive check (`!isConnected`) flashes an offline banner on first paint, because the websocket hasn't connected yet. That's wrong — the user just arrived; the browser hasn't even had a chance.

`shouldShowOfflineUi` suppresses the banner during an initial grace window and only goes true after a _real_ disconnect. That's the one you want in the layout.

::warning
Don't use `isConnected === false` on first hydration as an outage signal. You will ship a banner that flashes for everyone on page load. Use `shouldShowOfflineUi`.
::

## Safe to use in multiple places

The composable is backed by a shared store. Calling it in the root layout, in a global banner, and in a page-local indicator doesn't start three parallel subscriptions.

```ts [app/components/SyncIndicator.vue]
const { pendingMutations } = useConvexConnectionState()
```

```vue [app/components/SyncIndicator.vue]
<template>
  <span v-if="pendingMutations > 0" class="text-xs text-muted">
    Syncing {{ pendingMutations }}…
  </span>
</template>
```

## Hook into phase changes

If you want to react to the transition (toast on reconnect, log to telemetry), there's a Nuxt hook:

```ts [app/plugins/connection-telemetry.ts]
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.hook('trellis:connection:changed', (payload) => {
    // payload.state: 'connecting' | 'connected' | 'reconnecting'
  })
})
```

It fires on phase changes — not on every internal runtime update. Safe to log.

## Good use, bad use

| Good                                                          | Bad                                                             |
| ------------------------------------------------------------- | --------------------------------------------------------------- |
| Offline banner                                                | Deriving auth state from connection                             |
| Reconnect spinner                                             | Using `isConnected` as authorization                            |
| Disabling destructive buttons when many mutations are pending | Treating first-paint disconnect as an outage                    |
| "Changes are still syncing" hint                              | Gating sensitive UI on `isConnected` instead of the appIdentity |

::caution
Connection state is a UX signal, not a security signal. A signed-in page with `isConnected: false` is still a signed-in page — the guard on the server decides what the user can do, not the websocket.
::

## Common pitfalls

- Showing an outage banner from `!isConnected` on first paint instead of `shouldShowOfflineUi`.
- Gating auth or authorization UI on connection state instead of the backend-owned appIdentity and guards.
- Treating `pendingMutations` as a server truth signal instead of a client-side UX hint.

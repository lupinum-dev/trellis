Click from a loaded list into a detail page and render the item immediately — no blank flash, no spinner, no "loading…" before the real detail query resolves. The item is already in memory from the list query; seed the detail page with it while the full result loads.

```vue [app/pages/tasks/[id].vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const props = defineProps<{ id: string }>()

const {
  data: task,
  isFromCache,
  cacheStatus,
} = await useCachedQuery(api.tasks.get, () => ({ id: props.id }), {
  from: {
    query: api.tasks.list,
    args: {},
    find: (tasks) => tasks.find((t) => t._id === props.id),
  },
})
</script>
```

The detail page renders from the list match, then transparently upgrades to the real detail result when it arrives.

## What this is (and isn't)

`useCachedQuery` is not a general `useConvexQuery` replacement. It's a narrow helper for one pattern: **list → detail navigation when the list already contains enough of the item to stand in for it**.

::note
For everything else — fetching fresh data, reactive args, subscriptions — keep using [`useConvexQuery`](/docs/data-fetching/queries). This page covers one specific UX win.
::

## The flash it kills

Without cache seeding, the list-to-detail sequence is:

1. List page renders, item is visible.
2. User clicks into detail.
3. Detail page renders empty.
4. Detail query resolves.
5. Content appears.

Steps 3-4 are the flash. When the list item already has the fields the detail header needs, there's no reason to show an empty page between the click and the resolved data.

## `isFromCache` — render cues during the handoff

`isFromCache` is `true` while the page is showing the seeded value, `false` once the real detail query settles. Use it to show a subtle "refreshing" hint, not a full loading state:

```vue [app/pages/tasks/[id].vue]
<template>
  <header>
    <h1>{{ task?.title }}</h1>
    <span v-if="isFromCache" class="text-xs text-muted">Syncing…</span>
  </header>
</template>
```

## `cacheStatus` — did the seed actually match?

`cacheStatus` tells you what happened before the real detail query settled:

- `matched` — the list data contained a seed candidate
- `match-missing` — the list data existed, but `find(...)` returned nothing
- `source-missing` — there was no cached list data to seed from

That makes it much easier to debug the “why didn’t this seed?” case without guessing whether the list cache was missing or the match function drifted.

## What the seed depends on

The seed is pulled from Nuxt data for the parent list query. That means **the `from.query` and `from.args` have to match the query that populated the list page**.

If `from.find(...)` returns `undefined`, the helper transparently degrades to a regular query. Trellis warns when cached source data exists but the match misses, because that usually means `from.args` or `from.find(...)` drifted. Deep-links with no cached list data stay quiet and simply show `cacheStatus: 'source-missing'`.

## When this is worth it

| Fit  | Pattern                                                                                |
| ---- | -------------------------------------------------------------------------------------- |
| Good | Task board → task detail; project list → project detail; article list → article detail |
| Bad  | List and detail use unrelated queries                                                  |
| Bad  | List items only have an `_id` — nothing meaningful to show                             |
| Bad  | User deep-links straight to detail (no list to seed from — fine, it degrades)          |

## Common pitfalls

- Reaching for `useCachedQuery` on every detail page. Use it when there's an actual list-to-detail transition; otherwise you're paying complexity for nothing.
- Forgetting the seed is temporary — treat `isFromCache: true` as "this will upgrade," not as the final state.
- Mismatched `from.args` or a stale `from.find(...)` implementation. Check `cacheStatus`; `match-missing` means the list cache exists but your seed lookup no longer matches it.
